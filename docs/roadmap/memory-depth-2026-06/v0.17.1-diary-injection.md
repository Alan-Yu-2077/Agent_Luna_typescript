# v0.17.1 — Diary injection + monthly diaries (the long-range layer)

> **Status: PLANNED.** Initiative: Memory depth correction (Order 10, version 2/2). Risk: **Medium**.
> Depends: v0.17.0 (composable `buildActiveContext`). Flag: `LUNA_DIARY_INJECT` (default off → on after
> quality check).

## Goal

Make the diary / weekly / monthly summaries — the design's **long-range narrative memory** — actually
reach the model. Today they're written by the dream cycle and even embedded by `rag_refresh`, but **no
context-assembly path reads them**, so their contribution to what Luna knows is zero. This version wires
them back in two complementary ways (standing digest + recall candidates), generates the missing monthly
diary, and closes the owner's correction by amending LD #12's diary part. After this, Luna has "这几天 /
这几周发生了什么" continuity, not just the recent window + discrete facts.

## What ships

### Injection path 1 — a standing diary digest in the system block
- A short, dream-updated **"recent narrative" digest** (last few days + current week, optionally last
  month) rendered into the cached system block alongside core prose (`renderCoreBlock.ts`), so it's
  **always present** like the self-state/relationship prose. Bounded length (a few hundred tokens);
  refreshed by the dream cycle, not per turn (keeps the prefix cache stable — it only changes when the
  dream writes a new diary).
- New reader of the `diaries` table on the context side (today the only reader is `dream/cycle.ts`):
  a `renderDiaryDigest()` that pulls the latest day/week(/month) summaries.

### Injection path 2 — diaries as recall candidates
- Add `diaries` as a third candidate source in `collectCandidates` (`recall/recall.ts:111`) — extend
  `Candidate.source` from `'l2' | 'l3'` to include `'diary'`. This makes the embeddings `rag_refresh`
  **already computes** (`dream/cycle.ts:214`) finally retrievable (fixes the dead-work finding M6), so a
  query semantically near a past diary surfaces it.
- Per-hit rendering/truncation consistent with L2/L3 hits (`RETRIEVAL_K`, 300-char slice).

### Recall ranking — recency × importance × relevance (Generative Agents)
- Upgrade `retrieve()` scoring (today pure cosine relevance, `recall.ts:174`) to the canonical companion
  formula **`α·recency + β·importance + γ·relevance`** (Park et al. "Smallville"): blend the existing
  cosine *relevance* with a *recency* decay over the candidate's age and the *importance* signal defined
  in v0.17.0 (the same score that anchors compression). This is what keeps salient/recent diaries +
  L2/L3 hits ranked above stale-but-similar ones, and it's the same score reused — not new machinery.
- Default weights configurable; start with the GA baseline (all weights equal) and tune.

### Monthly diaries (close the gap)
- `dream/cycle.ts run_diaries` gains a `'month'` branch (the prompt already supports it,
  `dream/prompts.ts:71,77`) — generate a monthly retrospective on the appropriate cadence (e.g. when a
  month's worth of weeklies exist), same insert path as day/week (`:176,200`).

### Decision amendment (close the initiative)
- `REWRITE_CONTEXT.md` — amend **LD #12**'s diary part: diaries are an **injected long-range memory
  layer** (standing digest + recall candidate), not merely an L2 archive output. Record the rationale
  (owner correction, PR #3).
- Mark `docs/MEMORY_DESIGN_DIVERGENCE.md` diary rows ✅ done.

## Tests
- `renderDiaryDigest` — pulls latest day/week/month; bounded length; empty-safe (no diaries yet → empty
  block, no crash); only changes when a new diary is written (cache-stability assertion).
- `collectCandidates` — includes `'diary'` candidates; a query near a known diary retrieves it (golden);
  `Candidate.source` union updated; existing L2/L3 retrieval unchanged.
- `dream/cycle.test.ts` — a `'month'` diary is generated when the cadence condition holds; day/week
  unchanged; `rag_refresh` embeddings for diaries are now actually scored (end-to-end: write diary →
  refresh → retrieve).
- system-prompt test — the diary digest is present in the cached block and the block stays byte-stable
  across turns until a new diary lands (extends the byte-stable-prompt invariant).
- `tsc` clean; `bun test` green.

## What this version explicitly does NOT include
No change to *how* diaries are produced (the dream cycle's day→week→month generation stays; only monthly
is added and the read side is wired). No window-size change (v0.17.0). No new dream cadence engine beyond
the monthly trigger.

## Risk
**Medium.** The standing digest adds always-on tokens to the (cached) system block — bounded, and
cache-stable since it only changes on dream write, but size it deliberately. Adding `'diary'` to recall
candidates enlarges the candidate set → confirm it doesn't blow the recall latency budget (ties to
Initiative 9 P1's budget and the vec0 decision — a bigger corpus may warrant vec0 KNN). The monthly
generation must be idempotent (don't regenerate the same month). Watch: don't double-inject (a diary
appearing both as standing digest AND a recall hit is mild redundancy — dedup by period_key if it bites).

## Acceptance criteria
- [ ] Day/week(/month) diaries reach the model via a standing digest in the system block, dream-updated,
      cache-stable; tested.
- [ ] Diaries are retrievable recall candidates; `rag_refresh`'s diary embeddings are actually scored
      (no more dead vectors); tested end-to-end.
- [ ] Monthly diaries are generated by the dream cycle (idempotent); tested.
- [ ] `REWRITE_CONTEXT.md` LD #12 diary-part amended (diary = injected long-range layer);
      `MEMORY_DESIGN_DIVERGENCE.md` closed.
- [ ] `tsc` clean, `bun test` green, DEVELOPMENT.md v0.17.1 entry, **Initiative 10 closed**.

## Notes for after Initiative 10 (don't foreclose)
With diaries injected, the candidate corpus is materially larger — this is the natural moment to revisit
Initiative 9's vec0 decision (Open Q #2) in favor of wiring `vec0` KNN. Keep `renderDiaryDigest`
composable so a future "memory timeline" UI can render the same digest the model sees.
