# v0.17.0 — L1 window → ~100 clean turns + structured compression

> **Status: PLANNED.** Initiative: Memory depth correction (Order 10, version 1/2). Risk: **Medium**
> (down from High — the clean-history foundation makes the deep window cheap). Depends: Initiative 9 —
> **v0.16.3 (clean durable history)**, v0.16.1 (memoized system block), v0.16.2 (incremental
> `history_json`). Flag: `LUNA_L1_RECENT_TURNS` (the restored turns-unit knob, default ~100).

## Goal

Fix "她记得太短" at the conversational layer by replacing the 24-*message* (~4–9-turn) cap with a
**~100-turn verbatim window of clean conversation**, plus a **structured, bounded compression** of older
history beyond it. This is the within-recent-span half of the memory gradient (the cross-day half is the
diary, v0.17.1). It's affordable — *not* because we accept a huge prompt, but because v0.16.3 already
stripped thinking + collapsed tool I/O, so a stored turn is ~200 tokens and ~100 turns is ~20k tokens.
Decided with the owner after reviewing SOTA designs (Letta/MemGPT, Mem0, Generative Agents): the field
gets *depth* from a gradient (recent verbatim + compressed gist + importance anchors), not from a giant
verbatim buffer — so we keep a generous-but-sane verbatim window and let compression + diaries carry the
rest.

## Design rationale (from the SOTA review + the short-reply token math)
- **Luna's replies are short** (`message` caps at 280 chars / 5 sentences), so a *clean* turn is ~200
  tokens — 50–250× smaller than a code-agent turn. The aggressive-eviction numbers in the literature
  (SWE-agent "last 5 observations", JetBrains "~10 turns") are from large-IO tool agents and **do not
  bind here** → a ~100-turn verbatim window is comfortably affordable (~20k tokens).
- **The ceiling is quality, not cost**: context rot (recall accuracy decays as tokens grow — mild at
  20–30k, real past 100k+) and diminishing returns past ~100 recent turns. So ~100 sits in the sharp
  regime; the env knob allows 40–150 without a redeploy.
- **Depth ≠ verbatim count**: per Mem0's LOCOMO result, distillation beats full-context on cross-day
  recall *and* is cheaper — so beyond the recent window we **compress**, we don't extend verbatim.

## What ships

### The verbatim window, re-sized + re-unitized (`memory/l1Window.ts`)
- Replace `KEEP_MSGS = 24` (messages) with `LUNA_L1_RECENT_TURNS` (**turns**, default **~100**, range
  40–150), where a turn = one user message + its (clean, per v0.16.3) assistant reply group. `planFold` /
  `windowLowWater` count **turn-groups**, not raw messages — reversing the unit change PR #3 identified
  (`l1Window.ts:6,45`).
- Because turns are clean (v0.16.3), the window holds ~100 real exchanges at ~20k tokens.

### Structured oscillating compression (replaces the broken `rolling_summary`)
- Beyond the verbatim window, compress older turns into a **structured, bounded** summary — *not* the
  current single `rolling_summary` string that grows unboundedly on every fold (the real bug behind
  "lossy + ever-growing"). Structure it as buckets (key facts / decisions / open threads / emotional
  beats) with a hard size cap, so older history contributes a stable-size gist that covers well past 100
  turns of *coverage* at a fixed token cost.
- **Importance anchors** (borrowed from Generative Agents' importance score): high-importance turns
  resist compression — they stay verbatim or near-verbatim longer than the recency cutoff would imply, so
  the idiosyncratic, salient moments aren't sanded into generic gist (the documented over-summarization
  pathology). A cheap importance signal (e.g. an LLM 1–5 rating at fold time, or heuristic: named
  entities / first-person disclosure / emotional markers) gates this.

### Persistence (leans on v0.16.3 + v0.16.2)
- The persisted window is the clean ~100-turn tail + the structured summary; full history rebuilds from
  L2 on load (v0.16.2). Storing a 100-turn *clean* window is fine precisely because of v0.16.3.

### Decision amendments (same change set)
- `REWRITE_CONTEXT.md` LD #12 — active window = **~100 verbatim turns** (env-tunable 40–150), unit
  *turns*, with structured-bounded compression + importance anchors for older history.
- v0.4.1 plan's "12 pairs" default + messages unit → marked **superseded**.

## Tests
- `l1Window.test.ts` (rewritten, turns unit): a K-turn session keeps the last `LUNA_L1_RECENT_TURNS`
  verbatim and compresses the rest; the knob moves the boundary deterministically; fold groups by turn.
- compression: the older-history summary is **structured + size-capped** — repeated folds do **not** grow
  it unboundedly (the explicit regression vs today's `rolling_summary`); golden on a fixed transcript.
- importance anchors: a flagged-high-importance old turn stays verbatim past the normal cutoff; a mundane
  one is compressed; deterministic with an injected importance signal.
- token-bound assertion: a ~100-turn clean window assembles to ~20k tokens (golden, approximate); the
  assembled `MessageParam[]` is well-formed.
- **cost/latency measurement** recorded in the version entry (input tokens + TTFT at 24-msg vs 100-turn).
- `tsc` clean; `bun test` green.

## What this version explicitly does NOT include
No diary injection (→ v0.17.1) — this is the recent-conversation + older-compression layers only. No
change to *how* the clean turn is built (that's v0.16.3). No new recall behavior.

## Risk
**Medium.** The cost concern is largely retired by v0.16.3 (clean turns) + the measurement-before-flip
discipline. The two real risks: (1) the **structured compressor** must be bounded + faithful — get it
wrong and older history either bloats (inherits the `rolling_summary` growth bug) or over-summarizes
(erases the personal specifics that make her feel like she knows you) — pinned by the size-cap regression
test + importance anchors; (2) the prefix cache must not be churned — the window lives in the message
stream, and within a session its older portion is stable, so place the cache breakpoint to keep the
stable head cached (confirm no per-turn full re-send beyond the new tail).
