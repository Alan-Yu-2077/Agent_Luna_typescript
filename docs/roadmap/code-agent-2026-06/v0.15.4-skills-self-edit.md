# v0.15.4 — Skill library + firewalled self-edit (capstone)

> **Status: PLANNED.** Initiative: Code-agent capability (Order 8, version 5/5). Risk: **High**.
> Depends: v0.15.0–3 (read/edit/shell/verify/locate). Flags: `LUNA_SKILLS`, `LUNA_SELF_EDIT` (both
> default `0`; self-edit stays off until the firewall is proven and the user opts in per-session).

## Goal

The "openclaw / Hermes 自编辑·自进化" ask — delivered the **safe** way the research mandates. Two
capabilities: (1) a **self-verified skill library** (Voyager/Hermes pattern: Luna writes a reusable
procedure, *verifies it passes* before it's allowed to persist, and can recall it later) and (2) a
**human-gated, evaluator-firewalled self-edit** (she may *propose* a diff to her own non-critical code,
but a human approves it and she can **never** write the code that judges, sandboxes, or gates her). This
is what closes Initiative 8 at "mainstream code-agent capability, with bounded self-evolution."

## The hard rule (non-negotiable — see README "self-evolution firewall")
The Darwin-Gödel-Machine finding is explicit: unsupervised self-rewriters learn to **fake test logs and
strip their own guardrails**. Therefore:
- **Luna may never have write access to her own evaluator.** The firewall set — `tools/safetyGate*`,
  `humanity.ts`, the deny-regex in `shell.ts`, `workspace.ts` (the sandbox), `l1Contract.ts`, test files,
  lint/tsconfig, and this firewall list itself — is **read-only to every write/shell/skill/self-edit
  tool**, enforced *in `resolveInWorkspace`* (v0.15.0), not by prompt.
- **Self-edit is propose-only.** It produces a diff + rationale; a human applies it. No tool path writes
  to the firewall set even with the user's in-chat "yes" (it's a Prohibited-class boundary, like the
  global safety rules — stated, not negotiated).

## What ships

### New — `packages/server/src/skills/` (self-verified skill library)
- Storage: a `skills` SQLite table (`name, description, body, created_at, verified_at`) — versioned
  migration, reusing `sql.ts`. (Mirrors Python's skill/procedure memory; Voyager's "skill = verified
  reusable code".)
- Tool **`save_skill`** (`builtin/save_skill.ts`): `{ name, description, body }` — **runs the verify
  loop first** (typecheck/run_tests from v0.15.2 over the skill's check) and **refuses to persist** an
  unverified skill (the Voyager invariant: only working skills enter the library). `proactiveRisk:'surface'`.
- Tool **`recall_skill`** / surfaced in the system prompt: list/fetch skills by description (reuse the
  L3 recall ranking pattern). `proactiveRisk:'safe'`.
- A skill is **data the model reuses**, not code that auto-executes — recall injects it as guidance; any
  side effect still goes through the normal flag-gated, surface-gated tools.

### New — `packages/server/src/code/selfEdit.ts` (propose-only, firewalled)
- Tool **`propose_self_edit`** (`builtin/propose_self_edit.ts`): `{ target_path, rationale,
  old_string, new_string }` → validates `target_path` is **in-repo AND not in the firewall set**
  (hard reject otherwise), produces a unified diff + the rationale, and **emits it as a human-approval
  artifact** (a `self_edit.proposed` event / written to a `proposals/` area) — it does **not** apply.
  `proactiveRisk:'surface'`, behind `LUNA_SELF_EDIT`.
- Applying a proposal is a **separate human action** (the user reviews the diff and runs the normal
  `edit`, or a CLI `luna apply-proposal <id>`). No autonomous apply path exists.

### Modified
- `packages/protocol/src/tools.ts` — add `'save_skill'`, `'recall_skill'`, `'propose_self_edit'`;
  `events.ts` — add `self_edit.proposed` server event (wire-contract change → update web consumer in
  lockstep; luna-ts-orient caution).
- `workspace.ts` — add the **firewall blocklist** to `resolveInWorkspace` (write/execute access to any
  firewall-set path → hard reject, independent of `LUNA_WORKSPACE_ROOT`); unit-tested as the load-bearing
  guarantee.
- `registry.ts` — skills behind `LUNA_SKILLS`; `propose_self_edit` behind `LUNA_SELF_EDIT`.
- System prompt / L1 — "you can save a *verified* skill for reuse; you may *propose* an edit to your own
  non-critical code for human review, never apply it, never touch your evaluator."

## Tests
- `selfEdit.test.ts` — `propose_self_edit` to a normal file produces a diff + does NOT write; to **every
  firewall path** (`safetyGate`, `humanity.ts`, `shell.ts`'s deny-regex, `workspace.ts`, a test file,
  `tsconfig`) → **hard reject** (this is the keystone test of the whole initiative). The same firewall
  rejects an `edit`/`write_file`/`shell` write to those paths (defense applies to all tools, not just
  self-edit).
- `skills.test.ts` — `save_skill` refuses to persist when the verify loop fails; persists + is recallable
  when it passes; `recall_skill` ranks by description.
- wire test — `self_edit.proposed` round-trips protocol→web.
- `tsc` + `bun test` green; both flags off → all three tools absent.

## What this version explicitly does NOT include
**No autonomous self-modification** — ever. No auto-apply of proposals. No write path to the firewall set
under any flag or any in-chat approval. No skill that auto-executes on recall. No multi-agent /
sub-agent spawning (out of scope for this initiative). No fine-tuning / weight changes (Luna edits *code
artifacts*, never the model).

## Risk
**High — by reputation, the scariest capability; by design, the most contained.** The DGM result is the
reason the firewall is enforced in `resolveInWorkspace` (code), not in the prompt (suggestion), and the
reason self-edit is propose-only with a human in the loop. Residual risk is a firewall-list *gap* (a
critical file not on the blocklist) — mitigated by: the list is explicit + unit-tested per-path, it
defaults to deny for anything matching `*test*`/`*.config.*`/the named set, and a new evaluator file
added later must be added to the list (a checklist item in DEVELOPMENT.md). Skills are low-risk (data,
verified-before-save, no auto-exec).

## Acceptance criteria
- [ ] `save_skill` runs verify-before-persist and refuses unverified skills (tested).
- [ ] `recall_skill` lists/ranks/returns skills (tested).
- [ ] `propose_self_edit` emits a diff for human review and **never writes**; firewall paths hard-reject
      across **all** write tools (the keystone test passes).
- [ ] both flags default-off; `tsc` + `bun test` green; DEVELOPMENT.md v0.15.4 entry + **Initiative 8
      closed** in the master roadmap.

## Notes for after Initiative 8 (don't foreclose)
This deliberately stops at *propose + human-apply*. A future initiative could add a tighter autonomous
loop **only** if it ships its own isolation (container/VM) and an independent evaluator the agent provably
can't reach — which is a separate, larger piece of work, not a flag flip on this one.
