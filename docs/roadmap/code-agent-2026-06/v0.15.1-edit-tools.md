# v0.15.1 ÔÇö Edit tools (str_replace-native + fuzzy fallback)

> **Status: PLANNED.** Initiative: Code-agent capability (Order 8, version 2/5). Risk: **High**.
> Depends: v0.15.0 (workspace sandbox, `content_hash`, read-tracking seam). Flag:
> `LUNA_CODE_WRITE` (default `0` until E2E-verified, then flipped on).

## Goal

Give Luna **hands that can change code** ÔÇö the Claude-native edit surface plus a safe new-file write,
with the two reliability levers SOTA agents converge on: **read-before-edit** (no edits from stale
memory) and **lint/typecheck-on-write** (a broken edit is caught at edit time, not three turns later).
This is the riskiest *capability* (it mutates the user's files), so it's flag-gated, jailed, and
`proactiveRisk: 'surface'`.

## What ships

### New tools ÔÇö `packages/server/src/tools/builtin/`
All `concurrency: 'session-serial'`, `proactiveRisk: 'surface'`, paths via
`resolveInWorkspace(_, 'write')`, each returning a **unified diff** + new `content_hash`.

- **`edit`** (`edit.ts`) ÔÇö `{ path, old_string, new_string, replace_all?, expected_hash? }`. The
  Anthropic `text_editor` / Claude Code `Edit` shape. Rules:
  - **read-before-edit**: reject (recoverable, actionable) if `path` wasn't `read_file`'d this session
    (the v0.15.0 read-tracking seam). Forces the model to look before it leaps.
  - **uniqueness**: if `old_string` matches Ôëá1 time and `replace_all` is false Ôćĺ recoverable error
    *"old_string found N times ÔÇö add surrounding context or set replace_all"*.
  - **fuzzy fallback** (port of Python `edit_file` `_find_edit_match`, `filesystem.py:1279-1450`):
    exact Ôćĺ stripped-line (whitespace-tolerant) Ôćĺ `difflib`-style best-match hint on miss; **CRLF
    preserved**. When a non-exact match is used, **say so** in the result so the model can verify.
  - **optimistic concurrency**: if `expected_hash` given and the file changed Ôćĺ `stale_file` error.
- **`multi_edit`** (`multi_edit.ts`) ÔÇö `{ path, edits: {old_string,new_string,replace_all?}[],
  expected_hash? }`. Applies the sequence **atomically** (all-or-nothing; Claude Code `MultiEdit` /
  Python `patch_file:1453-1592`); on the first failed hunk, nothing is written + the failing index is
  reported.
- **`write_file`** (`write_file.ts`) ÔÇö `{ path, content, create_dirs?, overwrite?, expected_hash? }`.
  Full-file create/overwrite (port of Python `write_file:1113-1186`). Tool description **discourages**
  it for existing files (prefer `edit`); refuses to clobber without `overwrite`.

### New ÔÇö `packages/server/src/tools/lintOnWrite.ts`
After any successful `edit`/`multi_edit`/`write_file` to a `.ts`/`.tsx`/`.js`/`.jsx` file, run a fast
syntactic/type check and **fold diagnostics into the tool result** (SWE-agent ACI). v1: a fast parse
(oxc/`Bun.Transpiler` for a syntax check) ÔÇö full `tsc` is the heavier `typecheck` tool in v0.15.2.
Behind `LUNA_LINT_ON_WRITE` (default on once write ships). Diagnostics are *surfaced, not auto-reverted*
in v1 (reject-broken-edit is a v0.15.2 hardening option).

### Modified
- `packages/protocol/src/tools.ts` ÔÇö add `'edit'`, `'multi_edit'`, `'write_file'`.
- `packages/server/src/tools/registry.ts` ÔÇö mount the three behind `LUNA_CODE_WRITE` (registry chooses
  at boot, like `LUNA_MESSAGE_TOOL`).
- `packages/server/src/tools/dispatcher.ts` or session ÔÇö the read-before-edit tracking set (which files
  were read this session), populated by `read_file`, consumed by `edit`/`multi_edit`.
- L1 contract (`l1Contract.ts`) ÔÇö "after editing code, verify (typecheck/test); read a file before you
  edit it."
- `decision`-trace (optional) ÔÇö "promised to edit but didn't" reuses the `defectionAudit.ts` pattern.

## Tests
- `edit.test.ts` ÔÇö exact replace; uniqueness error on >1 match; read-before-edit rejection; fuzzy
  whitespace-tolerant match reports `fuzzed:true`; CRLF preserved; `expected_hash` mismatch Ôćĺ `stale_file`;
  unified diff correct; out-of-workspace path rejected.
- `multi_edit.test.ts` ÔÇö atomic: a failing 2nd hunk leaves the file untouched + reports the index.
- `write_file.test.ts` ÔÇö create with `create_dirs`; refuse overwrite without flag; `content_hash` returned.
- `lintOnWrite.test.ts` ÔÇö a syntactically broken edit returns diagnostics in the result.
- `tsc` + `bun test` green; with `LUNA_CODE_WRITE=0` the tools are absent (registry test).

## What this version explicitly does NOT include
No `shell` (v0.15.2) ÔÇö so directory create/move/delete (LD #9: folded into `shell`) are **not** here.
No full `tsc`/`bun test` verify tools (v0.15.2). No reject-broken-edit hard guard (v0.15.2 option). No
repo map (v0.15.3).

## Risk
**High** ÔÇö first tool that writes the user's files. Mitigations: default-off flag E2E-verified before
flip; jailed + sensitive-path-blocked (v0.15.0); read-before-edit + uniqueness + optimistic
`expected_hash` make a wrong-target edit hard; atomic `multi_edit` prevents half-edited files; unified
diff in every result makes changes auditable; `proactiveRisk:'surface'` blocks silent edits in proactive
turns. Watch: fuzzy matching must **report** when it fuzzed (a silent wrong-fuzz is the dangerous case).

## Notes for v0.15.2+ (don't foreclose)
The lint-on-write hook is the seam where v0.15.2 can add **reject-broken-edit** (SWE-agent style). The
read-tracking + diff plumbing is what `shell`'s "edited then ran tests" loop will lean on.
