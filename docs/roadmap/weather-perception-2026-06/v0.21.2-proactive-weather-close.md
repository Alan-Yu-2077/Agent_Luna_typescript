# v0.21.2 — Proactive weather in the opening + close

> **Status: PLANNED.** Initiative: Weather perception (Order 14, version 3/3). Risk: **Medium**.
> Depends: v0.21.0 (snapshot client), v0.21.1 (the cached `getSnapshot()` + `weatherAware` plumbing).
> Flag: `LUNA_WEATHER_PROACTIVE` (default-off → **flipped on at close**).

## Goal
Close the loop the owner asked for: when Luna **reaches out after a night** (a morning / after-overnight
wake), let her open with a **natural, ignorable** weather note — "morning — looks grey and cold out
there, layer up before you head out" — never a forecast recital, never on a routine afternoon
continuation. Then measure that the cache invariant held and **default-flip A/B/C on** to complete
Initiative 14.

## What ships
**The "after-a-night opening" signal (`turn/temporalContext.ts`, exported helper).**
- `afterANightOpening(nowMs, lastInteractionMs, tz): boolean` built from the **existing** helpers
  (`temporalContext.ts:217-229`): `daypart === 'morning'` (`classifyDaypart`, `:138`) **AND**
  `localDayNumber(now) !== localDayNumber(last)` (calendar-day crossing) **AND** `classifyGap(gapSec,
  crosses) ∈ {'new_day','long_away'}` **AND** `gapSec ≥ LUNA_NIGHT_MIN_GAP_SEC` (default ~6h, to exclude
  a trivial cross-midnight blip — there is no such gate in `classifyGap` today, `:171-172`). No new
  arithmetic — pure composition of shipped functions; `lastInteractionMs` sourced restart-safely from
  `listRecentL2().t_ms` as `runTurn` already does (`runTurn.ts:238`).

**The bounded weather note (`proactive/proactiveTurn.ts` `framing()`).**
- A `weatherNoteFor(snapshot, daypart): string | null` (pure, stateless, gated by
  `Bun.env['LUNA_WEATHER_PROACTIVE']`) reading the **same cached** `getSnapshot()` from v0.21.1 — it
  **never fetches** (proactive turns are socket-less and must not block on I/O). It returns a short,
  ignorable suggestion (or `null`).
- Inject it in `framing(intent, session)` (`proactiveTurn.ts:47-59`) **after** the felt-absence clause
  (`:51-58`) so a long-away morning wake can carry both; gate on daypart recomputed inside `framing()`
  (matching `feltAbsenceFor`'s recompute-from-time discipline) — fire it primarily when
  `afterANightOpening` is true (and optionally a lighter form on other morning wakes). It rides the
  **opening framing only** — it touches **neither** the wake *decision* (`wakeGate`), the cadence
  governor, nor the anti-repeat list.

**The "care, not forecast" guardrail (`persona/l1Contract.ts`).**
- Extend the `WEATHER_CLAUSE` (or add a proactive sentence) making explicit: weather in an opener is
  *warmth/care* ("bundle up", "perfect day for a walk"), a **suggestion you may ignore**, never a status
  report and never a guilt/obligation lever — the warmth-not-guilt sibling. Stays data-free + byte-stable
  (cache invariant), gated by the flag, folded into the memo key (`l1Contract.ts:52`).

**Measure + default-flip (close).**
- Confirm prompt-cache hit-rate is unchanged with all three layers on (the pin test stays green;
  weather lives only in the uncached tail + the data-free clause).
- **Default-flip** `LUNA_WEATHER` / `LUNA_WEATHER_AMBIENT` / `LUNA_WEATHER_PROACTIVE` from `=== '1'`
  (opt-in) to `!== '0'` (default-on) — the Initiative-12 close pattern — once verified end-to-end with a
  real `LUNA_LAT_LON`. Log `docs/history/DEVELOPMENT.md`; **Initiative 14 ✅**.

## Tests
- `temporalContext.test.ts`: `afterANightOpening` true for an overnight (≥6h) gap that crosses midnight
  into the 06:00–11:59 window; false for an afternoon continuation, a same-day short gap, and a sub-6h
  cross-midnight blip; `'long_away'` (≥24h) into a morning counts.
- `proactiveTurn.test.ts`: with `LUNA_WEATHER_PROACTIVE=1` + a stubbed morning `afterANightOpening` +
  a non-null cached snapshot, `framing()` appends the weather note **after** the felt-absence clause and
  **does not** invoke any fetcher; on a non-morning wake (or `null` snapshot) it appends nothing; the
  wake decision / anti-repeat list are untouched.
- Cache-invariant pin stays green with A+B+C on; a manual proactive morning wake produces a natural
  opener (not a forecast), observed once before the flip.

## What this version explicitly does NOT include
- Any change to the wake **decision**, cadence, or anti-repeat logic (LD #15 semantics unchanged) — only
  the opening **framing** gains a computed suggestion.
- A live fetch from the proactive path — it reads v0.21.1's cached snapshot only.
- Bi-directional / forecast-narration features — weather stays an ambient warmth lever.

## Risk
- **Medium** — design-sensitive, not technically risky. The danger is **over-mention** (the same
  over-fixation the subjective-time layer flagged): Luna narrating weather too often reads as a weather
  bot. Mitigation: gate hard on `afterANightOpening` (morning + real overnight gap), keep the note
  bounded + ignorable + suggestion-not-directive, and ship the `LUNA_WEATHER_PROACTIVE=0` kill switch.
  Observe in real use and tune the gap threshold / frequency (the `LUNA_TIME_SUBJECTIVE` precedent).

## Acceptance criteria
- [ ] On a morning/after-overnight wake Luna **may** open with a natural, ignorable weather note;
      on a routine afternoon continuation she **does not** mention weather.
- [ ] The note reads as **care, not forecast**; the proactive path triggers **no** network fetch.
- [ ] Cache hit-rate unchanged (pin test green); A+B+C **default-flipped on**; full suite green;
      `tsc` clean ×3; logged in `DEVELOPMENT.md`; **Initiative 14 ✅**.

## Notes (initiative close)
- With this version, weather is a complete sensory channel mirroring time perception: a pull tool (A),
  passive ambient awareness in the uncached tail (B), and a bounded proactive opener (C) — all
