# v0.22.2 — Event hooks + debounce + weather-shift

> **Status: PLANNED.** Initiative: Proactive pipeline redesign (Order 15, version 3/4). Risk: **Medium**.
> Depends: v0.22.1. Flag: `LUNA_PROACTIVE_EVENT_HOOKS` (default `1` after one release; behind-flag at first).

## Goal

Make the highest-value openings fire at their **natural instant**, not on the 60 s grid, by calling
the same `evaluateDetectors` funnel from real events — and add a `weatherShift` detector plus a
per-detector in-memory **debounce** so events can't double-fire.

## What ships

**`proactive/detectors.ts`** — add `weatherShift`: diff `getSnapshot()` against an in-memory baseline
(condition/temperature bucket); fires on a notable change with `debounceKey = "weather:<bucket>"`.
Baseline is in-memory (resets on restart — degrade OK, just means "no shift detected" until the next
refresh repopulates it). No network (reads the already-cached snapshot).

**Per-detector debounce** — an in-memory `Map<sessionId, Map<debounceKey, lastFiredMs>>` consulted in
the funnel: a `debounceKey` that fired within its window is skipped. Complements the persistent
slot-consumed bit (that one is per-day; this one is short-window, for event-driven re-evaluation).

**Event hooks (the same `safetyRail` → `evaluateDetectors` → single-turn-lock path):**
- **`ws.ts`** — on reconnect (`handleOpen`), if `afterANightOpening` is true for the session, fire one
  evaluation immediately (so the morning greeting lands the instant she's seen, not up to 60 s later).
- **Weather refresher** (`tools/web/weather/snapshot.ts` background refresh) — on a notable snapshot
  change, fire one evaluation for active sessions.
- Both funnel through a small shared **`maybeFireProactive(sessionId, ctx)`** — the one entry point the
  scheduler tick **and** the hooks call. It enforces a **real per-session single-turn guard**: a
  synchronous in-flight boolean flipped **before the first `await`**, so two callbacks racing on
  `activeTurn === null` can't both proceed (the existing check-then-act is **TOCTOU, not a lock** —
  review HIGH). `safetyRail` + `isDreaming` are applied once, inside the guard.

**`continuation.ts`** — route it through the **same single-turn guard** so a continuation and a
scheduler/hook turn can never overlap. Continuation stays deliberately **rail-light**: it is a bounded
*one-per-reply* micro-wake (LD #11), so it remains **quota- and cooldown-exempt by design** — which
means the cadence rail caps the **detector/scheduled** path, **not** continuation. (The README is
worded to say the rail caps the detector path, not "every proactive path" — see review HIGH.)

## Tests

- `detectors.test.ts`: `weatherShift` fires on a bucket change vs. baseline, not on noise; null when
  the snapshot is cold/unchanged.
- debounce unit test: a `debounceKey` fired within the window is skipped; a different key passes.
- `scheduler.test.ts` / a new `eventHooks.test.ts`: `maybeFireProactive` from a hook obeys the same
  `safetyRail` + single-turn lock; a hook firing while a turn is active or while dreaming is a no-op;
  a hook + a concurrent scheduler tick never run two proactive turns at once.

## What this version explicitly does NOT include

- Fuzzy detectors (`openThreadAged`, `promisedFollowThrough`) → v0.22.3.
- Deleting `wakeGate` → v0.22.3.
- Any cross-session fan-out (single-user today; the loop already iterates `activeSessionIds()`).

## Risk

- **Concurrency**: the hooks add new entry points → they MUST share a **real** single-turn guard, not a
  TOCTOU `activeTurn === null` re-read (`ticking` is scheduler-internal). A synchronous in-flight flag
  set **before the first `await`** is the primitive; `{tick, hook, continuation}` all acquire it.
  Double-fire is the main hazard.
- **Weather-shift noise**: bucket the diff coarsely (e.g. condition class + temp band) so ordinary
  fluctuation doesn't trigger; debounce caps frequency.
- In-memory baseline + debounce reset on restart — acceptable (worst case: one extra eval after a
  restart, still gated by the rail + daily quota).

## Acceptance criteria

- [ ] Reconnecting after an overnight gap fires the greeting **immediately** (not on the next 60 s tick).
- [ ] A notable weather change triggers at most one proactive eval (debounced), rail permitting.
- [ ] No two proactive turns run concurrently — `{tick, hook, continuation}` all acquire the **same
      synchronous single-turn guard** (not the TOCTOU `activeTurn` check).
- [ ] `bun test` green, `tsc` ×3 clean; `LUNA_PROACTIVE_EVENT_HOOKS=0` disables the hooks cleanly.

## Notes for v0.22.3 (don't foreclose)

- `maybeFireProactive` is now the universal entry point — fuzzy detectors slot into the same registry
  with no new plumbing.
- Once detectors fully cover the openings, `wakeGate` + `LUNA_PROACTIVE_LLM_GATE` have no remaining
  caller → safe to delete in v0.22.3.
