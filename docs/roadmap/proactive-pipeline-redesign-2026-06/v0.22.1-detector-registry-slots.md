# v0.22.1 — Detector registry + scheduled slots (a guaranteed speaking floor)

> **Status: PLANNED.** Initiative: Proactive pipeline redesign (Order 15, version 2/4). Risk: **Medium**
> (one SQLite migration). Depends: v0.22.0. Flag: `LUNA_PROACTIVE_SLOTS` (e.g. `'11,20'`; unset = off).

## Goal

Turn the inline MVP detector into a clean, extensible **registry**, and add a **scheduled-window**
detector so Luna has a non-zero daily speaking floor even when no content trigger fires (the antidote
to *under*-firing). Split the cadence prefilter into a reusable `safetyRail` + a `considerProactive`
that consults the registry.

## What ships

**`proactive/detectors.ts` (NEW)** — a `ProactiveDetector` type and an ordered registry:
- `type ProactiveDetector = { name: string; detect(ctx: DetectorCtx): { intent: ProactiveIntent;
  seed: string; debounceKey: string } | null }` — pure, no LLM, no network; reads only in-memory
  session + SQLite + cached snapshots; injectable `nowMs` (testable like `proactiveWeatherNote`).
- `afterNight` — lifts v0.22.0's inline check (wraps `afterANightOpening`).
- `scheduledWindow` — fires when the local hour ∈ `LUNA_PROACTIVE_SLOTS` and that slot is not yet
  consumed today; `debounceKey = "slot:<hour>"`.
- `evaluateDetectors(ctx)` runs the registry **first-match-wins by priority**, returns the first hit
  or `null`.

**`proactive/cadence.ts`** — split `shouldConsiderWake` into:
- `safetyRail(c, x) → { ok: boolean; reason }` — the **anti-spam** subset: **quiet hours / cooldown
  (5 m) / daily quota** only, plus a small **idle floor** (don't fire within ~60 s of the user's last
  message, so an event-hook detector can't cut into a live exchange).
  - **Drops the blanket `deep_absence` (> 18 h)** — a long absence is now a *trigger* (the after-night
    detector), not a suppressor (review HIGH). If any "truly gone, don't pile on" ceiling is ever
    wanted, make it **very high + detector-specific** (e.g. an env'd `> 7 d`), never a blanket pre-cut.
  - **Drops the `[10m,18h]` idle-window / `too_soon` *consideration* gate** — "consider" now means "a
    detector fired", not "idle 10–18 h". **Correction:** the old per-60 s re-fire came from a `hold`
    never stamping the cooldown, **not** from the idle window or the `too_soon` floor — and the
    detector model removes the speculative hold entirely, so it's moot.
- Add the per-day **slot-consumed** state to the `Cadence` type + persistence.

**Migration `migrations/0013_proactive_detectors.sql`** — add `proactive_slots_used` (INTEGER bitmask)
+ `proactive_slots_date` (TEXT) to `sessions`; `loadCadence`/`saveCadence` read/write them. Bitmask so
multiple slots/day are tracked in one column; resets when `slots_date` rolls over (local date, same
clock as quiet-hours/quota).

**`proactive/scheduler.ts` (`tickOnce`)** — replace the inline detector with: `safetyRail` →
`evaluateDetectors(ctx)` → on a hit, the TOCTOU re-check → `runProactiveTurn({intent, seed})` →
spoke/silent commit + mark the detector's `debounceKey` consumed (slot bit set) so it can't refire.

## Tests

- `detectors.test.ts` (NEW): table-driven, injected `nowMs` — `afterNight` fires only on the first
  post-overnight eval; `scheduledWindow` fires at a configured hour once/day and not again that day;
  empty `LUNA_PROACTIVE_SLOTS` → `scheduledWindow` never fires; registry first-match-wins ordering.
- `cadence.test.ts`: `safetyRail` blocks on quiet hours / deep-absence / cooldown / quota and passes
  otherwise; slot bitmask set/reset across a date rollover; migration applies on a fresh DB.
- `scheduler.test.ts`: a consumed slot does **not** refire the same day; no detector → no LLM, no turn.

## What this version explicitly does NOT include

- **Event-hook** firing (ws reconnect / weather refresher) + per-detector in-memory debounce +
  `weatherShift` → v0.22.2.
- Fuzzy detectors (`openThreadAged`, `promisedFollowThrough`) → v0.22.3.
- Deleting `wakeGate` → v0.22.3.

## Risk

- **Migration discipline**: `0013` is additive (new nullable/defaulted columns), no backfill needed;
  `loadCadence` tolerates a pre-migration row (defaults). Test on a fresh + an existing DB.
- A scheduled slot during a quiet-hour or deep-absence is correctly suppressed by `safetyRail` (slot
  is consumed only when the turn actually fires, not when blocked — so it can still fire later that day
  if the rail clears). Verify the consume-on-fire ordering.

## Acceptance criteria

- [ ] `detectors.ts` registry is pure + clock-injectable; `evaluateDetectors` returns the first hit.
- [ ] With `LUNA_PROACTIVE_SLOTS='11,20'`, she fires at most once per slot per day, rail permitting.
- [ ] `safetyRail` preserves every anti-spam guarantee; no `[10m,18h]` auto-fire remains.
- [ ] Migration `0013` applies cleanly; cadence round-trips the slot bitmask.
- [ ] `bun test` green, `tsc` ×3 clean.

## Notes for v0.22.2 (don't foreclose)

- `evaluateDetectors(ctx)` is the single funnel — v0.22.2 calls it from event hooks too (ws reconnect,
  weather refresher), so keep it free of scheduler-only assumptions (pass `ctx`, not the tick).
- The `debounceKey` shape (`slot:<h>`, later `thread:<id>`, `weather:<bucket>`) is the seam the
  per-detector debounce Map keys on next version.
