# v0.3.0 — Anthropic interleaved tool-use end-to-end

> **Status: PLANNED.** Initiative: tool-spec-foundation (Order 1, version 3/3). Risk: **Medium-High**
> (first real LLM integration; interleaved SSE protocol; tool-use round-trip). Depends: v0.2.0.
> Flag: none (provider is required from this version on).

## Goal

First real LLM round trip. A WS client sends a user message; the server calls Anthropic with the v0.2
tools attached; if the model emits `tool_use`, the v0.2 dispatcher runs it; the result flows back to
the model; the final reply streams back to the client. **Token streaming does not stop when a tool
call happens** — that is the whole point of this version.

This is the version that proves the rewrite's two stated goals (latency + typed contract) deliver
the win the audit predicted.

## What ships

### Provider layer

`packages/server/src/provider/anthropic.ts`:

- Uses `@anthropic-ai/sdk` officially. No hand-rolled HTTP.
- `chatStream(request) → AsyncIterable<ProviderEvent>` where `ProviderEvent` is a typed projection of
  Anthropic's SSE events: `text_delta`, `tool_use_start`, `tool_use_input_delta`, `tool_use_complete`,
  `thinking_delta`, `message_stop`.
- Honors `interleaved_tool_use` header (or equivalent SDK option) so the SSE stream can contain
  multiple text segments interleaved with tool calls.
- Reads `ANTHROPIC_API_KEY` and `ANTHROPIC_BASE_URL` from env at boot. No third-party gateway hacks
  baked into code — if Alan wants `https://yunwu.ai`, it goes in `.env`. Single source of truth.
- Configures `claude-opus-4-8` (or whatever is set in `LUNA_MODEL` env) once at boot.

### Turn orchestrator

`packages/server/src/turn/runTurn.ts`:

- Input: user message + session state.
- Output: `AsyncIterable<ServerEvent>` — emitted on the WS as the turn progresses.
- Loop:
  1. Build request: system prompt (placeholder string for v0.3) + history (in-memory) + user msg +
     `tools` from registry.
  2. Open `provider.chatStream(request)`.
  3. For each `ProviderEvent`:
     - `text_delta` → emit `reply.token` on WS.
     - `tool_use_complete` → translate to `ToolCall`, hand to dispatcher, await `Result`, append
       `tool_result` to next provider call. Emit `tool.started → tool.progress* → tool.finished` to
       WS in parallel.
     - `message_stop` with no tool use → emit `turn.result`, exit loop.
     - `message_stop` with tool uses → re-enter at step 1 with appended history. Bounded loop
       (`MAX_TOOL_ITERATIONS = 8` for v0.3; raise later if real use shows it tight).
- **Streaming-on-tools**: text deltas keep arriving on the WS while tools run. The dispatcher's
  `tool.progress` events interleave with `reply.token` events on the same WS — no separate channel,
  no `time.sleep`, no blocking pacing.

### Wire contract additions

New `ServerEvent` variants in `packages/protocol/src/events.ts`:

- `turn.started { turn_id }`
- `reply.token { turn_id, text }`
- `turn.result { turn_id, text, finish_reason, usage }`

New `ClientEvent`:

- `chat.send { turn_id?, text }` — turn_id auto-generated if absent.

The `tool.*` events from v0.2 stay unchanged.

### Session state

In-memory only for v0.3. A `Session` object holds:
- `id: string` (default `'default'` for now)
- `history: Anthropic.MessageParam[]`
- `mutex` for `session-serial` tools

One WS connection = one session. Multi-session waits for v0.4 (SQLite persistence).

### What does NOT change from v0.2

- The tool registry. The three v0.2 tools (`time_now`, `read_file`, `remember`) are the entire tool
  surface in v0.3. The other 7 land **after** v0.3 — once the protocol round trip is proven against
  a real LLM, adding tools is mechanical.
- The `Result<T>` discriminator.
- The dispatcher.

## Tests

1. **Happy path streaming**: send `chat.send { text: "what time is it?" }`. Server emits
   `turn.started → reply.token+ → tool.started(time_now) → tool.finished(time_now) → reply.token+ →
   turn.result`. Assert event ordering.
2. **Interleaving**: a tool that yields 3 progress events while the model is still streaming text →
   `reply.token` events appear **before AND after** the `tool.progress` events (proves true
   interleaving, not just sequencing).
3. **Tool validation failure → model recovery**: send a prompt that makes the model call `read_file`
   with a path that fails Zod input validation. Assert the model receives `Result.err('validation_failed', ...)`
   structured, NOT a string starting with "Error", and produces a graceful textual reply.
4. **Iteration cap**: a pathological prompt that loops calling `time_now` repeatedly — assert the
   loop terminates at `MAX_TOOL_ITERATIONS` and emits a structured `turn.result` with
   `finish_reason: 'max_iterations'`.
5. **WS reconnect mid-turn**: open WS, send message, kill the WS while a tool is mid-execution.
   Assert the dispatcher's tool actually completes (no half-state) and the next WS connection on
   the same session sees the prior turn in history. (Establishes the property for v0.4.)
6. **Provider error fallthrough**: mock the provider to throw mid-stream. Assert the server emits
   `error` event on WS with a structured code, does NOT silently re-run the call, does NOT silently
   double-pay tokens. (Direct response to Python audit's "double token spend" issue.)

## Mocking strategy

Real Anthropic calls cost money and are slow. v0.3 tests use:
- A `MockProvider` implementing the `chatStream` interface, returning a scripted `AsyncIterable<ProviderEvent>`.
- One **opt-in** integration test (gated by `LUNA_INTEGRATION_TESTS=1`) that hits the real
  Anthropic endpoint with a deterministic seed prompt and asserts the basic event shape.

## What v0.3 explicitly does NOT include

- Memory persistence (history is in-process, lost on restart — v0.4).
- Persona / humanity guardrails (system prompt is a placeholder — v0.6).
- Reasoning rails / L1 / L2 (model gets no special framing — v0.8).
- Proactive (no idle handling — v0.10).
- Frontend (`agent-app.js` port — v0.12).
- Audio / Live2D / TTS (v0.12+).
- Multi-session (one WS connection = one session, `'default'` id — v0.4).
- Skill subsystem (cut entirely per REWRITE_CONTEXT).
- Tool approval gating (cut entirely per REWRITE_CONTEXT).

## Risk

Medium-High. Specific concerns:

- **Anthropic SDK API for interleaved tool-use SSE.** As of conversation date, the SDK supports
  streaming with tools, but `interleaved_tool_use` may require an explicit beta header. Verify SDK
  version supports it before locking the API surface.
- **Bun's fetch interaction with the SDK.** Bun's `fetch` has occasionally diverged from Node's for
  streaming bodies. If issues arise, the SDK accepts a custom HTTP client.
- **Tool dispatcher's `session-serial` lock under concurrent WS frames.** The lock is per-session;
  if v0.3's session is process-global it works trivially. The shape needs to be right so v0.4
  multi-session doesn't need a rewrite.
- **Iteration cap behavior.** A model that keeps requesting tools forever must terminate cleanly. The
  `MAX_TOOL_ITERATIONS=8` cap mirrors Python Luna's circuit breaker concept but as a single hard
  bound — no 5-rule god-class detector.

- [ ] First token reaches the client in <500ms after `chat.send` (subject to provider latency floor).
- [ ] `tool.progress` events appear on the WS stream **before** the corresponding `tool.finished`

## Notes for Initiative 2 (don't foreclose)

- The `Session` shape in v0.3 is in-memory; in v0.4 it becomes SQLite-backed. The interface should be
  `getSession(id) / saveSession(session)` so the swap is mechanical.
- `runTurn` should already accept a `Session` abstraction, not reach into a global state map.
- Anthropic's `Message` shape (history) is what we'll persist. Don't invent a parallel shape — store
  it as-is so we can replay turns straight back to the SDK in v0.4.
- The placeholder system prompt in v0.3 becomes the composed `persona + humanity + scene` block in
  v0.6. Wire it through a single `buildSystemPrompt(session)` function so the substitution lands in
  one place.
