## Goal

Stand up the smallest runnable system that proves the runtime + wire choices work:
**Bun + TypeScript + Zod + WebSocket + monorepo layout.** No agent logic. No LLM call. No tools yet.

A connected client can open a WS and exchange typed echo events. That's it.

### Protocol

In `packages/protocol/src/events.ts`:

- `ClientEvent = { type: 'ping', seq: number }`
- `ServerEvent = { type: 'pong', seq: number, server_time_ms: number } | { type: 'error', code: string, message: string }`

Both Zod-defined; types derived via `z.infer`. Parse on inbound, format-then-send on outbound. No
hand-written interfaces, no `as any`.

### Server

- Bun's native `Bun.serve({ websocket: { open, message, close } })`. No Express, no socket.io.
- On `open`: log connection, no auth.
- On `message`: `ClientEvent.safeParse` the inbound text. If invalid тЖТ send `error`. If `ping` тЖТ send
  `pong` with current `Date.now()`.
- On `close`: log.

### Tooling

- `bun install` works.
- `bun run --cwd packages/server src/main.ts` starts the server on `localhost:8787`.
- `bun test` runs (suite empty for now; we add one trivial test that imports the protocol package).
- No build step yet. Bun runs TS directly.

## Tests

Minimal but real:

1. `protocol/src/events.test.ts` тАФ Zod schemas parse a known-good `ping` and reject malformed input.
2. `server/src/ws.test.ts` тАФ start the server on a random port, open a WS client, send a `ping`,
   assert a `pong` with matching `seq` comes back within 100ms.

That's it. No coverage targets. The point of v0.1 is to prove the skeleton.

## Risk

Low. Bun's API is stable; the protocol surface is two event types. The one place to watch is the WS
test тАФ Bun's test runner has occasional flakiness around port-bound resources. If that bites, use a
random high port + `await` the `open` event before sending.
