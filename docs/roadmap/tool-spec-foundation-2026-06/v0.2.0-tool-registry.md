# v0.2.0 — Typed tool registry + `Result<T>`

> **Status: PLANNED.** Initiative: tool-spec-foundation (Order 1, version 2/3). Risk: **Medium**
> (this is the core contract — getting the shape wrong here propagates everywhere).
> Depends: v0.1.0. Flag: none.

## Goal

Define the tool contract in code. Three representative tools land end-to-end with **typed input,
typed output, discriminated `Result<T>`, declared concurrency policy, per-tool `summarize()`, and
streaming progress events**. No LLM in the loop yet — execution is driven by a test harness that
calls the registry directly.

This is the version that decides what every future tool looks like. Worth one extra design pass.

## What ships

### `packages/protocol` — tool contract types

New module `tools.ts`:

- `ToolName = z.enum([...])` — the closed set of legal tool names (10 entries planned).
- `ToolResult<T>` — discriminated union:
  - `{ kind: 'ok', data: T, summary: string }`
  - `{ kind: 'err', code: ToolErrorCode, message: string, recoverable: boolean }`
- `ToolErrorCode = z.enum(['tool_not_found','validation_failed','execution_exception','timeout','aborted'])`
- `ToolEvent = z.discriminatedUnion('kind', [progress, final])`
  - `progress: { kind:'progress', tool_name, call_id, payload: unknown }`
  - `final: { kind:'final', tool_name, call_id, result: ToolResult<unknown> }`

New wire events added to `ServerEvent`:
- `tool.started { call_id, tool_name, input }`
- `tool.progress { call_id, payload }`
- `tool.finished { call_id, result: ToolResult }`

### `packages/server/src/tools/`

```
tools/
├── defineTool.ts         ← the factory: takes spec, returns a registered Tool
├── registry.ts           ← Map<ToolName, Tool>; resolve, dispatch
├── dispatcher.ts         ← runs ToolCall[] respecting concurrency policy
└── builtin/
    ├── time_now.ts       ← representative: instant, safe-parallel, no I/O
    ├── read_file.ts      ← representative: I/O, safe-parallel, real summarize
    └── remember.ts       ← representative: session-serial (would touch state in v0.4)
```

### The tool definition shape

A tool is declared with `defineTool({...})`:

| Field | Type | Notes |
|---|---|---|
| `name` | `ToolName` literal | Compile-time checked against the protocol enum |
| `description` | `string` | Used in the LLM tool definition (v0.3) |
| `input` | `z.ZodType` | Source of truth for TS type + JSON schema + runtime validation |
| `output` | `z.ZodType` | Validated before wrapping in `Result.ok` |
| `concurrency` | `'safe-parallel' \| 'session-serial' \| 'global-serial'` | Dispatcher honors this |
| `timeoutMs` | `number` | Hard cap; structured `timeout` error on breach |
| `summarize` | `(out) => string` | Per-tool; no global cap, no global compactor |
| `execute` | `async function*` | Async generator yielding `ToolEvent`s; final event must be `kind:'final'` |

The generator-based `execute` is what makes streaming progress natural: a long-running tool yields
progress events as it works; the dispatcher forwards them as `tool.progress` wire events; the final
yield is the result.

### Dispatcher behavior

`dispatchToolCalls(calls: ToolCall[], ctx) → AsyncIterable<ToolEvent>`:

- Group calls by `concurrency`:
  - `safe-parallel` → fire concurrently, interleave events as they arrive.
  - `session-serial` → queue against a per-session mutex.
  - `global-serial` → queue against the process-wide mutex.
- Per call: run `input.safeParse` → if invalid, immediately emit `final` with
  `Result.err('validation_failed', ...)`. Don't invoke `execute`.
- Per call: wrap in `Promise.race` against `timeoutMs`; on timeout, abort the generator and emit
  `final` with `Result.err('timeout', ...)`.
- Uncaught exception → `Result.err('execution_exception', error.message, recoverable: false)`.
- All output validated against `output` schema before being wrapped — schema mismatch becomes
  `validation_failed` on the **output** side. Catches buggy tool implementations.

No `startswith('Error')`. No string heuristics anywhere. Failure is a structured value.

### The three representative tools

Picked because each exercises a different axis of the contract:

1. **`time_now`** — zero I/O, instant, `safe-parallel`. Proves the contract works for the simplest
   case. Output: `{ iso: string, unix_ms: number, tz: string }`.
2. **`read_file`** — disk I/O, `safe-parallel`, real `summarize` (clip to ~200 chars + "[N more lines]").
   Proves I/O + structured truncation.
3. **`remember`** — `session-serial`, no actual persistence yet (just in-memory `Map`), but
   exercises the discriminator pattern (input has `kind: 'fact' | 'preference' | 'moment'`).
   Proves the serial-lock path works and that one tool absorbs the four Python memory tools.

These three are deliberately the smallest set that proves the contract is sound. The other ~7
tools (search, write, edit, recall, shell, list_files, message) come **after** v0.3 once the
contract is exercised against a real LLM.

## Tests

Concrete contracts the test suite must enforce:

1. **Type-level**: `ToolName` enum and the registry's keys are identical (compile error otherwise).
2. **Validation**: passing bad input to `time_now` returns `Result.err('validation_failed', ...)`
   without invoking `execute`.
3. **Concurrency**: launching three concurrent `time_now` calls completes in less than 2× a single
   call (proves parallel). Launching three concurrent `remember` calls completes in **at least** 3×
   the per-call cost (proves serial).
4. **Timeout**: a tool that sleeps longer than `timeoutMs` returns `Result.err('timeout', ...)` and
   the generator is actually aborted (proven by a side-effect counter).
5. **Streaming**: a tool that yields 5 progress events emits exactly 5 `tool.progress` events plus 1
   `tool.finished`, in order, on the dispatcher's output stream.
6. **Output validation**: a tool whose `execute` returns something not matching its `output` schema
   produces `Result.err('validation_failed', 'tool output schema mismatch')`.

## What v0.2 explicitly does NOT include

- No LLM (that's v0.3).
- No memory persistence (`remember` is in-memory only).
- No `write_file`, `edit_file`, `shell`, `search_code` (they land after the contract is proven
  against a real LLM in v0.3).
- No mount logic (every defined tool is always available; gating comes in v0.3 only for `shell` if
  Q1 in `REWRITE_CONTEXT.md` resolves that way).
- No verifier hooks on `write_file` (it doesn't exist yet).

## Risk

Medium — the contract shape is load-bearing for everything downstream. Specific risks:

- **Async generator + Zod output validation interaction.** If a tool's `execute` returns a stream of
  partial-state objects, output validation only makes sense on the `final` event. The validator must
  run **inside** the generator wrapper at the `final` boundary, not against intermediate `progress`
  payloads.
- **Per-session mutex granularity.** v0.2 has no real "session" yet — use a stub `sessionId`
  parameter so `session-serial` becomes a real lock in v0.4 without a refactor.
- **Result discrimination with generics.** TypeScript struggles with `Result<T>` exhaustiveness when
  T is unknown. Likely shape: `ToolResult` is the wire type (unknown data), `Result<T>` is the
  internal type. Convert at the boundary.

## Acceptance criteria

- [ ] `defineTool` returns a typed `Tool` whose `input`, `output`, and concurrency policy are all
  enforceable at type level.
- [ ] The registry rejects defining two tools with the same name at compile time (literal-type union).
- [ ] All six tests above are green.
- [ ] A WS client (test harness) can send a synthetic `ToolCall` to the server and receive the full
  `tool.started → tool.progress* → tool.finished` event sequence with matching `call_id`s.
- [ ] No `as any`, no `as unknown`, no `startswith('Error')`, no `instanceof Error` failure detection
  in the tool path. (`instanceof` is fine for *building* the error; not for *deciding* an error happened.)
- [ ] `packages/protocol` builds and is consumed by `packages/server` with shared types intact.
- [ ] `docs/history/DEVELOPMENT.md` updated with v0.2.0 row.

## Notes for v0.3 (don't foreclose)

- The `execute: async function*` shape is exactly what Anthropic's interleaved tool-use SSE wants on
  the other side — each `tool.progress` event can ride the same SSE stream as model `text` deltas.
  The integration is mechanical if v0.2 gets the generator right.
- LLM-driven tool calls in v0.3 will arrive as `tool_use` blocks in the Anthropic SSE; the dispatcher
  converts them to `ToolCall` objects, runs them, and emits `tool_result` blocks back. The wire shape
  on the WS side is the same `tool.*` events from v0.2 — nothing changes.
