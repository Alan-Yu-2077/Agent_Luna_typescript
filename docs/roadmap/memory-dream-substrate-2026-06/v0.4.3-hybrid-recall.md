# v0.4.3 — Hybrid recall (sqlite-vec embedding + CJK-bigram lexical)

> **Status: SHIPPED 2026-06-12.** Initiative: memory-dream-substrate (Order 2, version 4/5). Risk: **Medium**
> (first `sqlite-vec` integration; embedding API on a third-party gateway). Depends: v0.4.2. Flag:
> `LUNA_MEMORY_EMBEDDING` (default on for TS — diverges from Python's default-off; `=0` → lexical only).

## Goal

Make memory *recallable by meaning*, not just by recency. One retrieval substrate over both L2
full-text turns and L3 facts, ranked by a **hybrid** score: `sqlite-vec` cosine (embedding-first,
since SQLite + `sqlite-vec` is the locked substrate) blended with CJK-bigram lexical (so Chinese and
exact-token queries stay robust, and recall works offline / on embedding failure). Ports Python
v0.54's unified L2+L3 retrieval; upgrades the storage from a flat `embeddings.json` cache to a real
vector index.

## What ships

### Dependency
`sqlite-vec` (loaded as a `bun:sqlite` extension). Confirm Bun extension-load path at build; if
loading is fiddly, the fallback is a vec table + manual cosine in SQL (Python's "lightweight first").

### Schema
`migrations/0005_vec.sql` — `l2_vec` / `l3_vec` virtual tables (`vec0`, dim per embedding model) keyed
to `l2_turns.id` / `l3_facts.id`. An `embeddings` content-hash cache table (SHA-256 → vector) so
re-embedding identical text is free (ports Python's cache idea, `semantic_embedding.py`).

### `packages/server/src/memory/recall/` (new)
- `embed.ts` — `embedTexts(texts[]) → vectors[]` via the embedding API. A **minimal ~30-LOC fetch
  client** against the embeddings endpoint — Anthropic has no embeddings API, so this is necessarily
  a second HTTP path. **This is NOT a resurrection of the cut `openai_compat` adapter**: that cut
  targeted the chat-provider path (broken `api_key_override`); chat stays Anthropic-SDK-only. Config
  from env (Python `.env` has these): `LUNA_EMBEDDING_MODEL` (default `text-embedding-3-large`),
  `LUNA_EMBEDDING_API_KEY`, `LUNA_EMBEDDING_BASE_URL`. Cache-misses only (hash check first). Batched.
- `lexical.ts` — keyword + **CJK bigram** sliding-window scorer with stop-words (port the approach
  from `semantic_retrieval.py` / `daily_timeline.py:30-75`). No API, always available.
- `recall.ts` — `retrieve(query, {k, scope}) → Hit[]` over L2+L3: embedding cosine (if enabled) +
  lexical, blended with a **recency boost** (the new-near-old tie-break Alan accepted), returns top-k.
  **Soft-deleted facts (`deleted_ms` set) are excluded** unless an explicit `asOf` time-travel
  parameter is passed. Mirrors `DailyTimelineLog.retrieve_relevant` branching on
  `LUNA_MEMORY_EMBEDDING`.
- `renderRecallBlock.ts` — renders the retrieval-ranked top-k as a **message-level** context block.

### Modified
- **Injection placement (the prompt-cache invariant — TS-only, Python never had this constraint):**
  the per-query retrieval block is injected at **message level** (a `<memory>` context block inside
  the user turn), **after** the cached prefix. The system prompt keeps only v0.4.2's stable
  `renderCoreBlock` and **never varies per query** — a retrieval-ranked block in the system prompt
  would invalidate the Anthropic prefix cache on every single turn, taxing exactly the latency the
  rewrite exists to win. `LUNA_MEMORY_RETRIEVAL_K` default ~12.
- v0.4.2's category-capped facts in the core block shrink to the durable highlights; the per-query
  detail moves to the recall block (clean division: system = who Luna is + what's always true;
  message = what's relevant right now).
- L2/L3 writes (v0.4.0/v0.4.2) — embeddings are **lazy** (computed on recall cache-miss) to keep the
  write path LLM/API-free; dream's `rag_refresh` (v0.5.0) pre-warms the cache in bulk.

## Tests
1. Semantic recall: store facts, query a paraphrase (no shared keywords) → embedding path returns
   the right fact; assert it ranks above an unrelated one.
2. CJK lexical (embedding off): a Chinese query returns bigram-matched L2 turns without any API call.
3. Hybrid + recency: two equally-similar facts, the newer ranks first (recency boost).
4. Cache: embedding the same text twice hits the hash cache (one API round-trip, asserted via a mock
   embed client call-count).
5. `LUNA_MEMORY_EMBEDDING=0` → pure lexical, no API calls, still returns sensible hits.
6. Integration: the recall block lands in the **user message**, and the system prompt stays
   byte-identical across queries (MockProvider snapshot assert — the cache invariant).
7. Soft-deleted facts never surface in recall; `asOf` time-travel returns them when valid at that time.

## What this version explicitly does NOT include
- Bulk pre-warming during dream (`rag_refresh`) — v0.5.0 (this version embeds lazily on recall).
- **A model-callable `recall` tool** (agentic memory search — "let me think back…"). Injection-only
  for v0.4.x. Parked deliberately, not dropped: revisit after LD #9 (message_tool, v0.6) lands, when
  the tool surface and the everything-as-tool interaction patterns are settled. Tracked as Open Q #9
  in REWRITE_CONTEXT.
- Graph / multi-hop relationship recall — future; the hybrid here is embedding + lexical + recency.
- A real external vector store (Pinecone etc.) — `sqlite-vec` is the substrate; revisit only on a
  measured scale wall (Python's v0.56 "optional vector-store upgrade — measure first").

## Risk
- **`sqlite-vec` under Bun** — extension loading is the unknown. Mitigation: prove a 10-line
  load+query spike before building on it; fallback to a vec table + SQL cosine if loading fights Bun.
- **Embedding API on yunwu gateway** — same gateway as the LLM; reuse the two-attempt key cascade
  pattern (Python v0.56.2) if `LUNA_EMBEDDING_API_KEY` hits a rate-limited group. Lazy embedding +
  cache keeps API pressure low.
