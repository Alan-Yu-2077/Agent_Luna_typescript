# v0.5.0 — Dream engine (manual + tool-triggered, isolated consolidation)

> **Status: PLANNED.** Initiative: memory-dream-substrate (Order 2, version 5/5 — the capstone).
> Risk: **High** (LLM-driven multi-step cycle; reconciliation correctness; inherits two yunwu
> failure modes). Depends: v0.4.3. Flag: dream is opt-in by construction (WS command / tool);
> **auto-trigger is explicitly out of scope** (deferred to post-Initiative-5).

## Goal

The capstone: an isolated background mode where Luna digests what happened, reconciles her long-term
facts, writes a diary, and updates her sense of self and the relationship — then parks asleep until
woken. Ports Python Luna's shipped dream cycle (`dream.py`, v0.55–v0.57). The live turn does **zero**
memory LLM work (that property already holds from v0.4.x); all consolidation is here, and it never
overlaps a live turn.

## What ships

### Wire contract (`packages/protocol/src/events.ts`)
- `ClientEvent` += `dream.enter {}` , `dream.wake {}`.
- `ServerEvent` += `dream.status { is_dreaming, current_step, last_dream_ms }`,
  `dream.step { step, status, detail }` (per-step progress, like Python's `DreamReport` steps).
- `chat.send` while `is_dreaming` → `error { code:'dreaming', message }` (the dream gate — Python's
  `_check_dream_gate` / `ConflictError`, adapted to the single WS: reject, never interleave).

### Tool
- `enter_dream { reason? }` added to `ToolName` + registry — Luna can choose to sleep
  (Python's `enter_dream`, rides alongside the memory tools). `session-serial`.
- **Mid-turn semantics (TS-only correction of a Python race):** the tool does **not** start the
  cycle during the live turn — it sets a `pending_dream` intent on the session and returns
  immediately. The cycle starts only after the current turn's `finalize` completes (checked in
  `runTurn`'s finally / the ws layer post-turn). Python's `enter_dream` started the daemon thread
  inside tool execution, leaving a tail-race window where early dream steps could overlap the
  turn's closing response — the TS isolation contract is absolute: **dream never overlaps any part
  of a live turn**, including its own trigger's turn.
- Tool-surface note: `enter_dream` brings the eventual surface to ~11 (the "~10" budget +1, spent
  consciously — sleep-on-her-own-judgment is core to the companion positioning, worth the slot).

### `packages/server/src/dream/` (new)
- `dreamState.ts` — SQLite-persisted state (`migrations/0006_dream.sql`: `dream_state` single row +
  `dream_reports` history). `is_dreaming`, `current_step`, `finished_idle` parked flag, `last_dream_ms`,
  `cycle_count`. Mirrors Python `dream_state.json` + the `finished_idle` parked-state semantics
  (v0.56.0): a completed cycle parks at `finished_idle` keeping `is_dreaming=true`; only `dream.wake`
  flips it.
- `cycle.ts` — the dream cycle is a **second StateGraph**, not a bespoke pipeline runner. The step
  order ports `DREAM_STEPS` (`dream.py:728-735`) but the execution shape is the TS line's own:
  `turn/graph.ts`'s `runGraph` is generalized over its node union (`runGraph<S, N extends string>` —
  a type-level refactor, the shipped turn loop is unchanged), and the six dream steps become nodes
  on a `DreamNode` union with typed transitions. Sequential if/else step-runners are exactly the
  shape this codebase rejected at v0.3 — dream gets the same graph treatment as turns, and per-step
  tracing comes free through the same `onTransition` seam instead of hand-rolled progress plumbing:
  1. `refine_semantic` — L3 de-dup / supersede / prune (calls `l3Store`; soft-delete semantics).
  2. `refine_layer1` — consolidated L1 summary from L2 full-text (supersedes the async fold).
  3. `memory_audit` — LLM reviews L3 vs recent dialogue → `forgetFact(stale ids)` (soft) +
     `addFact(new)`. **Reconciliation = supersede, never in-place UPDATE.**
  4. `persona_update` — LLM reviews core memory vs dialogue → `updateCore(prose patch)` (audited).
  5. `run_diaries` — backfill day / week / month diaries from L2 into a `diaries` table
     (3-tier, token-budgeted like Python 1400/1800/2400). Recent N injected by v0.4.x recall.
  6. `rag_refresh` — `embedTexts` over the union of new L2 + L3 text to warm the cache (v0.4.3 hook).
- **Trace identity**: the trace store is turn-scoped (`trace_id = turn_id`, per-turn buffer +
  flush). A dream cycle is not a turn — it traces under a synthetic `trace_id = dream:<cycle_id>`
  and **flushes per step**, not per cycle (a cycle runs minutes; a crash mid-cycle must not lose
  the completed steps' traces). Browsable at `/_trace` as its own tree.
- `llm.ts` — the dream LLM caller with the **two inherited production hardenings**:
  (a) prompts use **natural-language section headers**, never `<<<MEMORY_BEGIN>>>`-style delimiters
  (Python v0.56.1 — those trip yunwu's content filter);
  (b) **two-attempt key cascade**: try `LUNA_SUMMARIZER_API_KEY` first, fall back to the default key
  on empty/error; classify failures (`rate_limited` / `content_filter` / `auth` / `empty_text`)
  for the `dream.step` detail (Python v0.56.2).
  Implementation: **two `@anthropic-ai/sdk` client instances** (one per key) — the cascade is a
  client swap, never a hand-rolled HTTP path (chat-provider discipline applies to dream calls too).

### Modified
- `turn/graph.ts` — `runGraph` generalized over the node union (type-level only; turn loop untouched).
- `ws.ts` — `dream.enter`/`dream.wake` branches; `chat.send` gate; `enter_dream` tool routes here.
- `runTurn` — unchanged on the hot path (already LLM-free for memory); checks `pending_dream` after
  `finalize` to start a tool-requested cycle; the dream gate lives at the `ws.ts` ingress, not
  inside the turn loop.

## Tests
1. **Isolation / gate**: while a (mock) dream runs, `chat.send` returns `error{code:'dreaming'}`;
   no turn executes; after `dream.wake`, chat works.
2. **Parked state**: cycle completes → `is_dreaming` stays true, `current_step='finished_idle'`;
   `dream.wake` flips it; waking mid-step is rejected (`task_in_progress`).
3. **Reconciliation (the character-coherence test)**: plant a contradiction (L3 has "likes cats";
   dialogue says "got a dog, off cats"). After `memory_audit`, the stale fact is forgotten and the
   new one added — **never both coexisting**. Assert exactly one survives.
4. **Diary**: a day with L2 turns → a `diaries` day row; 7 days → a week row.
5. **Core memory update**: `persona_update` changes `self_state`/`relationship_status`, prior state
   in the audit table, `restore(1)` works.
6. **LLM hardening**: a mock that fails the summarizer key falls back to the default key; a delimiter
   in a prompt is asserted absent (natural-language headers only).
7. **rag_refresh**: warms only cache-misses; reports `misses_before/filled/after`.
8. **Trace**: a dream cycle traces under `dream:<cycle_id>` with **per-step flushes** — kill the
   process after step 3 and steps 1–3 are durable in the trace table; browsable at `/_trace`.
9. **Tool trigger isolation**: `enter_dream` called mid-turn returns ok, the turn completes normally,
   and **no dream-step trace carries a timestamp earlier than `turn.result`** (the pending-intent
   contract — no Python-style tail-race).

## What this version explicitly does NOT include
- **Auto-trigger** (idle threshold / too-long-since-dream safety net / wake-on-input) — **DEFERRED to
  after Initiative 5 (proactive)**, because it reuses the proactive idle scheduler (`_handle_idle`
  equivalent) that does not exist until v0.10. Python deferred its `v0.56.0-dream-autotrigger` for the
  same "prove the manual version first" reason.
- Frontend dream overlay / 🌙 button / Live2D sleep idle — Initiative 6 (frontend port, v0.12). v0.5.0
  is backend + wire contract; the manual trigger is the WS `dream.enter` event + the `enter_dream` tool.
- Cancel-mid-step — Python doesn't expose it either (`_cancel_flag` reserved, not wired to wake).

## Risk
- **Reconciliation correctness** is the load-bearing risk — a botched `memory_audit` either leaves
  contradictions or deletes good facts. Mitigate: supersede-only (forget+add), every change audited,
  test 3 guards coexistence, dream is low-frequency so a bad cycle is recoverable via the audit trail.
- **Multi-step LLM cycle on a gateway with content-filter + rate-limit traps** — both hardenings are
  ported from Python's actual incident fixes (v0.56.1/.2), not invented. Build them in from the start.
- **Isolation under single WS** — the gate must be airtight; a `chat.send` slipping through during a
  consolidation step is exactly the race the whole Python redesign existed to kill.

## Acceptance criteria
- [ ] Manual dream (WS `dream.enter` **and** `enter_dream` tool) runs all 6 steps and parks at
  `finished_idle`; `dream.wake` resumes.
- [ ] `chat.send` is rejected during dream — no interleaving.
- [ ] Planted contradiction is reconciled to a single surviving fact (test 3).
- [ ] Day/week diaries written; core memory updated + audited; rag cache warmed.
- [ ] Dream prompts carry no `<<<...>>>` delimiters; key cascade falls back correctly.
- [ ] Normal turns still make zero memory LLM calls.
- [ ] Full cycle is traced and browsable at `/_trace`.
- [ ] `bun test` green; `tsc` clean.
- [ ] `DEVELOPMENT.md` v0.5.0 row + record; initiative marked ✅ in both READMEs; Open Q #6/#7 struck.

## Notes for the deferred auto-trigger (don't foreclose)
- Keep `DreamCycle.start()` callable from outside `ws.ts` so the proactive idle scheduler (Initiative
  5) can invoke it. Persist `last_dream_ms` so the too-long-since-dream safety net can compute
  staleness. The auto path adds: idle-threshold entry, wake-on-input between steps, and the
  accumulated-turns fallback — all on top of the v0.5.0 manual core, no rewrite.
