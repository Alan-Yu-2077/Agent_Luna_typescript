## Goal

Keep the context window bounded as conversations grow, **without** the accumulating distortion that
re-compressing-the-compressed causes. Recent N turns go to the model verbatim; everything older is
folded **once** into a rolling summary and never re-compressed. When higher accuracy is needed, the
summary is re-derived from L2 full-text ‚Äî never from a prior summary. L2 (v0.4.0) is the only ground
truth; L1 is a bounded view of it.

## What ships

### Schema
`sessions` table gains `rolling_summary TEXT` + `window_low_water INTEGER` (the turn index up to
which content has been folded into `rolling_summary`). Migration `0003_l1_window.sql`.

### `packages/server/src/memory/l1Window.ts` (new)
- `buildActiveContext(session) ‚Üí MessageParam[]` ‚Äî returns `[summary-system-message?, ...recent N
  verbatim turns]`. N from `LUNA_L1_RECENT_TURNS` (default e.g. 12 pairs).
- `maybeFold(session)` ‚Äî when verbatim turns exceed `N + foldBatch`, fold the oldest unfolded turns
  (those between `window_low_water` and the new low-water) **from L2 full-text** into a single
  appended summary chunk; advance `window_low_water`. The fold reads L2, **not** the existing
  `rolling_summary` ‚Äî the summary only ever grows by appending freshly-derived chunks, never by
  re-summarizing itself.

### Modified
- `runTurn.build_request` ‚Äî instead of `session.history` verbatim, calls `buildActiveContext`. The
  full `history` stays in the `Session` object + L2; only what's *sent to the model* is windowed.
- `maybeFold` is **async fire-and-forget**, scheduled after `turn.result` is emitted (post-`finalize`,
  outside the turn). It never blocks the reply or the next turn ‚Äî **the live-turn hot path makes zero
  synchronous memory LLM calls** (the TS line's latency goal; this is also where the Python port gets
  corrected: Python ran the fold summary in an aux thread pool ‚Äî the async property is load-bearing
  and must not be lost in translation). Write protocol: the fold snapshots `window_low_water`, derives
  the new summary chunk from L2, then commits with a compare-and-swap (`UPDATE ‚Ä¶ WHERE
  window_low_water = :snapshot`) ‚Äî if a concurrent fold advanced it, this fold discards. A turn that
  starts while a fold is in flight reads the old summary + slightly more verbatim turns (soft window);
  both views are consistent because fold input is always L2, never the live history.

## Tests
1. **Bounded**: a 40-turn conversation sends ‚â§ `N + summary` messages to the (mock) provider, not 40.
2. **No re-compression drift** (the core test): fold turns 1‚Äì10 ‚Üí summary S1. Add turns, fold 11‚Äì20
   ‚Üí summary S2. Assert S2 was derived from L2 turns 11‚Äì20 **only** (the fold input never includes
   S1's text). Re-deriving S1 from L2 twice yields identical output (deterministic input).
3. **Re-derive from L2**: a helper that rebuilds the full rolling summary from L2 full-text produces
   stable output across two runs (ground-truth recoverability).
4. **`LUNA_L1_WINDOW=0`**: full history sent (v0.4.0 behavior).
5. **Fold never blocks**: with a fold in flight (mock summary call held open), a new `chat.send` is
   accepted and completes ‚Äî no `turn_in_progress`, no added latency. The fold's CAS commit then either
   lands or discards cleanly.

## What this version explicitly does NOT include
- L3 facts / core memory (v0.4.2).
- Recall/embedding (v0.4.3).
- Moving the fold into dream (a later refinement; v0.4.1 folds inline). Dream's `refine_layer1` step
  (v0.5.0) will own the heavier L1 summary; v0.4.1's inline fold is the lightweight always-on path.

## Risk
- **Compression drift** ‚Äî mitigated structurally: the fold input is L2 full-text, never the prior
  summary. This is the single most important invariant in the file; test 2 guards it.
- **Async fold vs concurrent turn** ‚Äî the race is real but bounded: the only shared write is
  `rolling_summary` + `window_low_water`, committed via CAS; readers get old-or-new, never torn.
  Test 5 guards the no-blocking property.
