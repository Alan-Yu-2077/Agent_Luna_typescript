# v0.23.3 — Per-model quirks + model registry + multi-model E2E (initiative close)

> **Status: PLANNED.** Initiative: OpenAI-protocol adapter (Order 16, version 4/4). Risk: **Medium**.
> Depends: v0.23.2. Flag: none new (consolidates the v0.23.1/v0.23.2 interim flags).

## Goal

Turn "an OpenAI provider exists" into "**Luna runs on any model**": a small **model registry** that
maps a model id → provider + endpoint + capability quirks, so picking a model is one config entry,
not a scatter of env flags. Verify across ≥2 distinct OpenAI-protocol models + Anthropic. Amend
`REWRITE_CONTEXT.md`, refresh the orient skill-map, close the initiative.

## What ships

- **`provider/registry.ts` (NEW):** `type ModelEntry = { id: string; protocol: 'anthropic' |
  'openai'; baseUrlEnv?: string; apiKeyEnv?: string; capabilities: Partial<ProviderCapabilities>;
  tokenParam?: 'max_tokens' | 'max_completion_tokens'; systemRole?: 'system' | 'developer';
  reasoning?: boolean }`. A built-in table for the common targets (Claude family → anthropic;
  GPT/o-series → openai+reasoning+`max_completion_tokens`+`developer`; generic OSS gateway model →
  openai defaults) + an override hook (`LUNA_MODELS_JSON` or a config file) so a new model is
  addable without a code change. `providerFor()` consults the registry: it resolves `LUNA_MODEL` →
  the entry → constructs the right provider with the entry's capabilities/quirks (replacing the
  v0.23.1/v0.23.2 interim flags `LUNA_OPENAI_REASONING`, ad-hoc token-param handling).
- **Per-model quirk handling in the OpenAI translate/request path** (driven by the entry, never a
  model-id regex at the call site):
  - token param: `max_tokens` vs `max_completion_tokens`.
  - system role: `system` vs `developer` (some reasoning models reject `system`).
  - `toolUse:false` models: omit `tools`, and the registry/`runTurn` surfaces a clear error if a
    tool-required turn targets a no-tool model (don't silently drop tools).
  - reasoning models: `reasoning_effort` mapping from the existing effort intent; `thinking:true`.
- **`main.ts` startup log:** print the resolved model id + protocol + key capabilities (replaces the
  Anthropic-only log line at [`main.ts:113-115`](../../../packages/server/src/main.ts)).
- **`REWRITE_CONTEXT.md`:** finalize the provider-LD amendment (multi-provider chat via a registry;
  the cut `openai_compat` rationale addressed); note the cache-invariant divergence on non-Anthropic
  models as an accepted trade.
- **`.claude/skills/luna-ts-orient/SKILL.md`:** add `provider/{factory,capabilities,registry,
  openai/*}.ts` + the `LUNA_PROVIDER`/registry flags to the map.

## Tests

- `provider/registry.test.ts`: `LUNA_MODEL` resolution → the right protocol + capabilities; an
  override entry (JSON) adds a new model; an unknown model id falls back to a safe OpenAI default
  (or errors clearly, per Open Question #2's resolution).
- `provider/openai/translate.test.ts` (extend): `developer`-role and `max_completion_tokens` quirks
  apply when the entry sets them; a `toolUse:false` entry omits `tools` and a tool-required turn
  raises a clear error.
- **Multi-model E2E** (manual, scripted like `scripts/proactive-soak.ts`): a full reactive turn
  (text + tool + `message`) on (a) a GPT/o-series model and (b) a second OpenAI-protocol model via
  the gateway, plus (c) Anthropic unchanged — all three persist correct history.
- Repo-wide: `LUNA_PROVIDER=anthropic` default path's full suite green; no model-id regex at any
  call site (grep guard).

## What this version explicitly does NOT include

- A provider-neutral message IR (the round-trip stays) — deferred per the README decision unless
  the round-trip proves lossy.
- A web-UI model picker / thinking-availability surfacing — only if Open Question #3 is answered yes
  (would be a follow-on, with a protocol-package wire addition).
- Embeddings/recall provider changes (LD #13 path unchanged).

## Risk

- **Medium** — breadth (per-model quirks) over depth. Mitigated by the registry centralizing every
  quirk behind `capabilities` (one place to fix a model), and by the multi-model E2E catching the
  real-backend variance the unit fixtures can't.
- Registry override format (env JSON vs file) is a small design choice — pick the simpler per Open
  Question #2; keep the built-in table authoritative so the common path needs no config.

## Acceptance criteria

- [ ] One registry entry (built-in or override) is all it takes to run a new OpenAI-protocol model;

- [ ] Per-model quirks (token param, system/developer role, no-tools, reasoning) are entry-driven,
      with **no model-id regex** at call sites.
- [ ] Multi-model E2E passes: GPT/o-series + a second OpenAI-protocol model + Anthropic, each a full
      tool-using turn with correct history.
- [ ] `REWRITE_CONTEXT.md` provider LD amended; `luna-ts-orient` map updated.
- [ ] **Initiative acceptance** (README) all checked; `bun test` green, `tsc` ×3 clean.
- [ ] Master `docs/roadmap/README.md` marks Initiative 16 ✅.

## Notes (initiative done)

- Adding a model is now a registry entry. If a future model needs a genuinely new wire shape
  (beyond Anthropic/OpenAI), it is a third `protocol` + provider behind the same `Provider` seam —
  the seam, capabilities, and registry make that additive.
- If the Anthropic⇄OpenAI round-trip ever shows loss (e.g. reasoning replay across providers), that
  is the trigger to revisit the deferred neutral-IR — file it as a new initiative, not a patch here.
