# v0.23.x — Post-ship audit: OpenAI-protocol adapter (Initiative 16)

> **Status: AUDIT (advisory).** Initiative: OpenAI-protocol adapter (Order 16, v0.23.0–v0.23.3,
> code-complete). Audited **2026-06-29**, before any live OpenAI run. Verdict: **no critical**,
> **5 high + 12 medium + 27 low/info**. The net-new code is careful; nearly every flaw is the same
> root cause — **treating "Anthropic's reliability" (forced tool use, prompt cache, SDK auto-retry,
> tolerant parsing) as protocol-independent** when porting onto the stricter, less-forgiving OpenAI
> Chat-Completions wire and the third-party gateways this adapter exists to support.

## Method

Seven independent review lenses (translation correctness · SSE streaming · provider↔consumer
contract · config/registry/factory · security · error handling · design/parity) fanned out over the
Initiative 16 diff and the consumer (`runTurn`), then **every** finding was adversarially
re-verified against the code by a separate skeptic (refute-by-default). 45 findings filed, 44
survived verification, 1 refuted. The highest-impact items were additionally re-confirmed by hand
against current code. Scope = Initiative 16 changes, including where a **pre-existing design now
interacts badly** with the new protocol.

| Severity | Count | Character |
|---|---|---|
| Critical | 0 | — |
| High | 5 | Default config dead-on-arrival · common model behavior silently fails · a slightly-off gateway response crashes the whole turn |
| Medium | 12 | Config footguns · reliability regressions vs the Anthropic path · history poisoning that 400s the next request |
| Low/Info | 27 | Resource leaks · cost regression · semantic loss · maintainability |

Affected files (all under [`packages/server/src/provider/`](../../../packages/server/src/provider/)):
`openai/openaiProvider.ts`, `openai/translate.ts`, `registry.ts`, `factory.ts`, `capabilities.ts`,
plus the consumer [`turn/runTurn.ts`](../../../packages/server/src/turn/runTurn.ts) and wiring in
[`main.ts`](../../../packages/server/src/main.ts).

## Theme A — message-tool mode on OpenAI is structurally fragile *(top design risk)*

message mode (Luna speaks **only** via the `message` tool; a free top-level text block is treated as
narration and not shown) was designed around Anthropic's reliable forced tool use. The port keeps
that assumption but drops the guarantee.

- **[HIGH] Content-only OpenAI reply in message mode is (almost) dropped.** Neither path sets
  `tool_choice` ([`openaiProvider.ts:165`](../../../packages/server/src/provider/openai/openaiProvider.ts#L165)).
  GPT-family models routinely answer in `content` instead of calling an offered-but-not-forced tool.
  In message mode top-level text is **not** surfaced ([`runTurn.ts:330`](../../../packages/server/src/turn/runTurn.ts#L330)),
  so the turn triggers the empty-reply guard → **one wasted full LLM round** → on a second
  content-only reply, a degraded fallback puts the leaked text into `turn.result.text`
  ([`runTurn.ts:546-619`](../../../packages/server/src/turn/runTurn.ts#L546)). Output is not strictly
  lost, but every such turn costs an extra round-trip and may not render as a bubble in the
  message-mode UI. *Trigger:* `LUNA_PROVIDER=openai` (or any `gpt-`/`o-` model) + default message mode.
  *Fix:* set `tool_choice: {type:'function', function:{name:'message'}}` (or `'required'`) when the
  `message` tool is present; and make the degraded fallback actually deliver a bubble, not just trace.
- **[MEDIUM] No `tool_choice` forcing** (root cause of the above):
  [`openaiProvider.ts:165-176`](../../../packages/server/src/provider/openai/openaiProvider.ts#L165).
  Luna's only voice is not guaranteed on the OpenAI backend.
- **[MEDIUM] SSE streaming is opt-in** (`LUNA_OPENAI_STREAM` default off):
  [`openaiProvider.ts:159-163`](../../../packages/server/src/provider/openai/openaiProvider.ts#L159).
  The default OpenAI experience has no token streaming and, in message mode, no live bubble preview —
  a visible UX regression vs Anthropic. *Fix:* default it on, or warn loudly at boot when off.

## Theme B — parsing is zero-tolerance toward the very gateways it targets *(robustness)*

The adapter exists to serve non-Anthropic-native gateways (the yunwu OpenAI route is the default
base), yet it is **strict where it should be tolerant and tolerant where it should be strict**.

- **[HIGH] `parseStreamChunk` / `parseOpenAIResponse` use Zod `.parse()` (throws).**
  [`translate.ts:141-143`](../../../packages/server/src/provider/openai/translate.ts#L141),
  [`translate.ts:235-237`](../../../packages/server/src/provider/openai/translate.ts#L235). A delta
  missing `index`, a partial `usage` shape, or any unmodeled-but-valid field → `ZodError` bubbles
  through `open_stream`, **crashing the whole turn** and discarding already-streamed text. The
  Anthropic SDK tolerates unknown fields by design. *Fix:* `safeParse` + skip-and-continue per chunk;
  relax non-load-bearing fields (`index` → `.optional().default(0)`).
- **[HIGH] Missing per-delta tool-call `id` → empty/colliding `tool_call_id`.**
  [`openaiProvider.ts:228-269`](../../../packages/server/src/provider/openai/openaiProvider.ts#L228).
  If a gateway echoes ids only on the final assembled message, `tool_use_start` never fires (no live
  preview) and `message_stop` emits `id=""`; the next request carries empty/duplicate ids, which
  OpenAI rejects (1:1 match required). *Fix:* synthesize a stable `call_${index}` id per index.
- **[MEDIUM] Mid-stream `{error:...}` frame is silently swallowed.**
  [`openaiProvider.ts:217-220`](../../../packages/server/src/provider/openai/openaiProvider.ts#L217).
  A streaming rate-limit/quota/server error (HTTP 200 + an error frame) parses to an empty chunk → the
  user gets a truncated/empty turn that may be persisted as a real short reply, poisoning recall. The
  non-streaming path checks `res.ok`; the streaming path does not. *Fix:* model `error` on the chunk
  schema and throw a descriptive error.

## Theme C — orphaned `tool_call` poisons history → next request 400s *(round-trip integrity)*

The SSE path **already** defends against this (forces `stopReason='tool_use'` when tool calls were
accumulated, [`openaiProvider.ts:258-262`](../../../packages/server/src/provider/openai/openaiProvider.ts#L258),
with a comment naming the exact failure). The buffered path — **the default** — does not. This is the
cleanest "definitely a bug," and an internal inconsistency between the two paths.

- **[MEDIUM] Buffered path + `finish_reason` `'length'`/`'stop'` while `tool_calls` present.**
  [`openaiProvider.ts:192-199`](../../../packages/server/src/provider/openai/openaiProvider.ts#L192)
  maps to `end_turn`/`max_tokens`, so [`runTurn.ts:369`](../../../packages/server/src/turn/runTurn.ts#L369)
  does **not** dispatch — but the `tool_use` blocks were already pushed into `session.history`
  ([`runTurn.ts:364`](../../../packages/server/src/turn/runTurn.ts#L364)). Next turn,
  `messagesToOpenAI` re-emits an assistant `tool_call` with no answering `tool` message
  ([`translate.ts:61-79`](../../../packages/server/src/provider/openai/translate.ts#L61)) → **400, the
  session wedges** until history is cleared. Same class reachable on a `length`-truncated tool turn.
- **[MEDIUM] Mid-turn variant** (filed `high`, **verifier-downgraded**): the originally-filed trigger
  ("`dispatch_tools` throws") is essentially unreachable — the dispatcher exhaustively swallows
  tool/mutex errors into error results
  ([`dispatcher.ts`](../../../packages/server/src/tools/dispatcher.ts), `mergeAsync` has no
  `onSourceError`). The real, reachable variant is the `finish_reason` mismatch above. Recorded for
  the fix to cover both.

*Fix (covers C):* mirror the SSE guard on the buffered path, **and** make rollback ordering-aware —
on any kept history tail, synthesize `is_error` `tool_result` blocks for unanswered `tool_use` ids (or
have `messagesToOpenAI` emit a synthetic error `tool` message defensively) so the wire request is
always well-formed.

## Theme D — default config footguns *(several make the OpenAI path dead-on-arrival)*

- **[HIGH] base-URL fallback sends the OpenAI route — and the bearer key — to the wrong path/host.**
  [`openaiProvider.ts:32-39`](../../../packages/server/src/provider/openai/openaiProvider.ts#L32):
  `LUNA_OPENAI_BASE_URL ?? ANTHROPIC_BASE_URL ?? api.openai.com/v1`. `ANTHROPIC_BASE_URL` is by
  convention the **bare host** (`https://yunwu.ai`; the Anthropic SDK appends `/v1/messages`), so the
  OpenAI route resolves to `https://yunwu.ai/chat/completions` — **missing `/v1` → 404**. Under the
  agent's dev env (`ANTHROPIC_BASE_URL=api.anthropic.com`) it ships the OpenAI request **+ bearer key
  to Anthropic's host**. In the documented default the provider is dead-on-arrival. *Fix:* don't fall
  back to `ANTHROPIC_BASE_URL`; require `LUNA_OPENAI_BASE_URL` (fail fast) or normalize a `/v1` suffix;
  log the fully-resolved `chatUrl` at boot.
- **[HIGH] `LUNA_PROVIDER=openai` + `LUNA_MODEL` unset → wire model is `gpt-4o-mini`, startup log says
  `claude-opus-4-8`.** Module-level `const MODEL = … ?? 'gpt-4o-mini'`
  ([`openaiProvider.ts:26`](../../../packages/server/src/provider/openai/openaiProvider.ts#L26)) is
  decoupled from the registry resolution in
  [`factory.ts:15-20`](../../../packages/server/src/provider/factory.ts#L15); the log prints the env
  default ([`main.ts`](../../../packages/server/src/main.ts)). Silent model substitution + a lying log.
  *Fix:* thread the resolved/forced model id from the factory into the provider as the single wire
  model; log the model that will actually be sent.
- **[MEDIUM] `Number(LUNA_MAX_TOKENS)` → `NaN` → serializes to `null` → 400.**
  [`openaiProvider.ts:27,143,171`](../../../packages/server/src/provider/openai/openaiProvider.ts#L27).
  One mistyped env var = total chat outage. *Fix:* `Number.isFinite(n) && n>0 ? n : 8192` (apply to
  [`anthropic.ts:12`](../../../packages/server/src/provider/anthropic.ts#L12) too).
- **[MEDIUM] `LUNA_MODELS_JSON` entry with an empty/short `id` becomes a silent catch-all.**
  [`registry.ts:20-27,60-62`](../../../packages/server/src/provider/registry.ts#L20). A typo reroutes
  **every** model (including Claude) onto the wrong protocol. *Fix:* `id: z.string().min(1)`; warn on
  very short prefixes that shadow a built-in family.

## Theme E — reliability/parity regressions vs the Anthropic path

- **[MEDIUM] OpenAI path has no retry** (`defaultFetch`/`defaultStreamFetch`,
  [`openaiProvider.ts:48-106`](../../../packages/server/src/provider/openai/openaiProvider.ts#L48)),
  while Anthropic uses `maxRetries:2` ([`anthropic.ts:54`](../../../packages/server/src/provider/anthropic.ts#L54)).
  One transient 502/503/429 fails the whole turn. *Fix:* a small bounded retry on connect/non-ok
  before first byte, honoring `Retry-After` — or document the divergence.
- **[MEDIUM] `complete()` can't disable reasoning → o-series dream/fold starves to empty.**
  [`openaiProvider.ts:147-157`](../../../packages/server/src/provider/openai/openaiProvider.ts#L147).
  Anthropic's `complete()` deliberately omits thinking so reasoning can't eat `max_tokens` and return
  empty; the OpenAI path has no equivalent. On `o1/o3/o4` (or reasoning `gpt-5`), L1 fold
  ([`l1Window.ts`](../../../packages/server/src/memory/l1Window.ts)) and the dream summarizer cascade
  ([`main.ts:71-78`](../../../packages/server/src/main.ts#L71)) can systematically return empty. *Fix:*
  send `reasoning_effort:'low'|'minimal'` for utility calls and/or a `max_completion_tokens` floor;
  consider a separate `LUNA_SUMMARIZER_MODEL`.

## Recommended remediation order (proposed v0.23.4 — "OpenAI hardening")

Smallest-change, highest-ROI first; the first two are **release blockers** for any OpenAI deployment.

1. **Theme D blockers** — drop the `ANTHROPIC_BASE_URL` fallback (require/normalize
   `LUNA_OPENAI_BASE_URL`, log resolved `chatUrl`); make the factory the single source of the wire
   model + fix the startup log; `NaN`-guard `MAX_TOKENS`; `id.min(1)` in the registry schema.
2. **Theme B** — `safeParse` + skip-bad-chunk in the SSE loop; synthesize `call_${index}` ids; detect
   `error` frames.
3. **Theme C** — mirror the SSE `tool_use` guard on the buffered path + ordering-aware rollback /
   synthetic error `tool` messages in `messagesToOpenAI`.
4. **Theme A** — `tool_choice` force the `message` tool in message mode; deliver the degraded fallback
   as a real bubble; decide the streaming default.
5. **Theme E** — bounded retry parity; reasoning guard in `complete()`.
6. **Theme F / tests** — error-body redaction, SSE reader cleanup + size caps, and tests for the
   riskiest paths (`complete()` reasoning starvation, abort, HTTP-error mapping, message-mode-without-forcing).

> All findings were produced **before a live OpenAI run** — verify against an actual gateway when
> `LUNA_PROVIDER=openai` is first exercised (cf. Initiative 15's "shipped but not live-tested" note).
