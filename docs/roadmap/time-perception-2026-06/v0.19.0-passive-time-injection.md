# v0.19.0 — A: passive time injection

> **Status: PLANNED.** Initiative: Time perception (Order 12, version 1/3). Risk: **Low–Medium**.
> Depends: nothing. Flag: `LUNA_TIME_AWARE` (default off).

## Goal

Stop Luna from drifting on time by **passively handing her the time every turn** — current local time,
day-of-week, daypart, **how long since the last message**, and a **session bucket** — all pre-computed in
TypeScript and placed in the **uncached** per-turn user message so the prompt cache is untouched. This is
the direct fix for the "called an hour-ago event 'yesterday'" bug: she never does the subtraction, so she
can't get it wrong. It also lays the time-helper module v0.19.1/v0.19.2 reuse.

## What ships

**New `packages/server/src/turn/temporalContext.ts`** — pure, TS-computed, no LLM:

- `classifyDaypart(hour: number): 'late night' | 'morning' | 'afternoon' | 'evening'` (Python's boundaries
  as the default: <6 / <12 / <18 / else).
- `formatGap(seconds: number): string` — humanized: `just now` / `12m` / `1h 12m` / `3h` / `2 days` …
- `classifyGap(gapSeconds: number, crossesCalendarDay: boolean): 'continuation' | 'same_day' | 'new_day' | 'long_away' | 'first'`
  — bucket from the gap **and** whether it crosses a local calendar day (so "this morning vs yesterday" is
  decided in code, never by the model). Thresholds env-tunable (`LUNA_TIME_*`).
- `buildTimeBlock(opts: { nowMs, lastInteractionMs: number | null, sessionStartMs: number, tz: string }): string`
  — returns the labeled block, e.g.:
  ```
  Current time (you are handed this — do not compute durations yourself):
  - Now: Wednesday, 2026-06-17 14:32 (Asia/Shanghai, UTC+8) — afternoon
  - Since the last message: 1h 12m (same day — still this afternoon)
  - This session: started 2h 03m ago
  ```
  First turn / unknown → `Since the last message: first contact`.

**`runTurn.ts` (build_request node)** — push the time block into the per-turn `blocks`, right before/after
the recall block and the `userText` text block (i.e. inside the `role:'user'` message — the **uncached
tail**), gated on `LUNA_TIME_AWARE`. **Never** added to `buildSystemPrompt` (the cached prefix).

- **Gap source:** the **last persisted L2 turn's `t_ms`** (`listRecentL2(sessionId, 1)`) for
  `lastInteractionMs`, so the gap survives a server restart; fall back to `session.lastUserMs`
  (`session.ts:31`) when no L2 row. `sessionStartMs` from a new non-persisted `Session.sessionStartMs`
  (init at `getSession`, boot time) — so "this session" resets per process (correct: a restart is a new
  session).

**`persona/l1Contract.ts`** — a `TIME_CLAUSE` pushed in `renderL1Contract` when `LUNA_TIME_AWARE` is on
(thread a `timeAware` arg through, memo key extended): *"You're handed the current time and how long since
the last message. Let it inform your tone and how you pick the conversation back up — you almost never need
to announce the clock or state an exact duration. Trust the handed labels; never compute 'how long ago'
yourself."* (Python's "don't announce" + the research's "don't self-compute.")

**Env (`.env.example`):** `LUNA_TIME_AWARE` (off), `LUNA_TIME_GAP_CONTINUATION_S` (600),
`LUNA_TIME_GAP_LONG_AWAY_S` (86400) — bucket knobs read live.

## Tests (`packages/server/src/turn/temporalContext.test.ts`, `bun test`)

- `formatGap`: 0→`just now`, 90→`1m`/`90s` (pick + assert), 4320→`1h 12m`, 180000→`2 days`.
- `classifyDaypart`: 2→late night, 9→morning, 14→afternoon, 21→evening.
- `classifyGap`: <600s same-clock → `continuation`; 2h same calendar day → `same_day`; 20h crossing midnight
  → `new_day` (NOT `same_day` even though <24h — the calendar-day flag decides); >1d → `long_away`; null → `first`.
- `buildTimeBlock`: contains the day-of-week + ISO date + tz offset + the `formatGap` string + the bucket
  label; first-contact path renders "first contact".
- **Placement (the cache-safety guarantee):** in a `runTurn` with `LUNA_TIME_AWARE=1`, the assembled
  **system block is byte-identical** to `LUNA_TIME_AWARE=0` (time is NOT in the cached prefix), and the time
  text appears in the latest `role:'user'` message. With the flag off, no time block anywhere.
- L1: the `TIME_CLAUSE` is present in the contract iff `LUNA_TIME_AWARE` on.

## What this version explicitly does NOT include

- **No relative-time labels on recalled memories** → v0.19.1 (B). A handles "now + gap"; B handles "when did
  *that* happen".
- **No subjective/felt-time, no proactive/dream integration** → v0.19.2 (C).
- **No default-on** — `LUNA_TIME_AWARE=0` ships; flip in v0.19.2 after measuring cache impact (expected nil).

## Risk

- **Prompt-cache regression** — the one real risk; mitigated by construction (time goes in the uncached
  user tail, never the system block) and proven by the byte-identical-system-block test. This is why A is
  Low–Med, not Low.
- **Over-fixation / false precision** — a model handed "02:47" may turn clipped or "worried". Mitigated by
  handing **bucketed labels** ("late night") + the L1 "don't announce" clause; tune in C.
- **Timezone off-by-one** — compute local time with an explicit offset/IANA zone in TS; never let UTC leak
  (a documented Claude Code bug). The block always states the zone.

## Acceptance criteria

- [ ] A TS-computed time block (now + daypart + elapsed + bucket) is injected in the per-turn user message
      under `LUNA_TIME_AWARE`; the cached system block is byte-identical with the flag on vs off (tested).
- [ ] Gap survives a restart (sourced from L2 `t_ms`); first-contact handled.
- [ ] The L1 time clause is present when on; `tsc` clean + `bun test` green; ships off.
