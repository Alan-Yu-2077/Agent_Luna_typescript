# v0.19.1 — B: memory temporal grounding

> **Status: PLANNED.** Initiative: Time perception (Order 12, version 2/3). Risk: **Medium**.
> Depends: v0.19.0 (the shared `relativeLabel`/`formatGap` helpers). Flag: `LUNA_RECALL_TIME_LABELS`
> (default off).

## Goal

Let Luna **place recalled memories in time**. Today the recall block hands her past L2 turns, facts, and
diaries with **no timestamp**, so she can't tell whether something happened an hour ago or last week — the
true root of the "yesterday" drift (which is a *dating-a-past-event* error, not a current-gap error that A
addresses). B renders a **relative-time label** on each recalled candidate (computed in TS from its `t_ms`)
and orders the candidates **chronologically**, so she reads "you mentioned the Heat Waves fetch — *this
morning*" instead of guessing. It reuses the existing Generative-Agents recall ranking; it only adds the
time *rendering*.

## What ships

**`packages/server/src/memory/recall/recall.ts`** — `renderRecallBlock(hits)` change (the read-side only;
`collectCandidates`/scoring untouched):

- Each candidate already carries `t_ms` (`recall.ts:85`, stamped for l2/l3/diary at `:95-126`). Render a
  **relative-time label** via the shared `relativeLabel(t_ms, nowMs)` from v0.19.0's `temporalContext.ts`:
  `just now` / `this morning` / `yesterday` / `3 days ago` / `last week` … with an **absolute date past a
  horizon** (default ~7 days → `on 2026-06-09`). The label is computed in TS — the model never derives it.
- **Chronological order**: within the recall block, present candidates **oldest→newest** (or newest-first,
  decided at build) and tag each with its label, e.g.:
  ```
  Things you might be recalling (each tagged with when it happened — trust these, don't recompute):
  - [this morning] you fetched the Heat Waves lyrics for Alan
  - [yesterday] Alan said the crack is open and you can use the new tools
  - [last week] …
  ```
  The ordering matters because shuffled-time facts measurably degrade temporal reasoning (Test of Time:
  ~28-point drop from fact-order shuffling) — so a stable chronological presentation is itself a mitigation.
- Keep the recall **ranking** by the GA score (`α·recency+β·importance+γ·relevance`) for *selection*; the
  chronological order is for *presentation* of the selected set. (Select by relevance/recency/importance,
  then sort the chosen few by time for display.)

**`memory/diaries.ts` `renderDiaryDigest`** — the standing day/week/month digest already labels entries by
`period_key`; add the relative phrasing (`[today] / [this week] / [3 weeks ago]`) for consistency with the
recall labels (gated on the same flag).

**Env:** `LUNA_RECALL_TIME_LABELS` (off), `LUNA_RECALL_ABS_DATE_DAYS` (7 — the relative→absolute horizon).

## Tests (`bun test`)

- `relativeLabel`: a `t_ms` 2 min ago → `just now`/`a few minutes ago`; today 08:00 from an afternoon `now`
  → `this morning`; 1 calendar day ago → `yesterday`; 3 days ago → `3 days ago`; 10 days ago → an absolute
  date (`on YYYY-MM-DD`), per `LUNA_RECALL_ABS_DATE_DAYS`.
- `renderRecallBlock` with `LUNA_RECALL_TIME_LABELS=1`: each rendered candidate line carries its `[label]`;
  the lines are in chronological order; with the flag off the block is unchanged (back-compat).
- A diary digest entry carries the relative label when on.
- Determinism: labels are a pure function of `(t_ms, nowMs)` — no model call, no nondeterminism.

## What this version explicitly does NOT include

- **No change to recall *selection*/scoring** — the GA ranking (Initiative 10) stands; B is presentation
  only. (A recency-decay re-tune, if ever wanted, is separate.)
- **No subjective/felt time** → v0.19.2 (C).
- **No bi-temporal "valid vs learned" timestamps** (Zep/Graphiti) → deferred option D; B uses the single
  creation `t_ms` only.
- **No default-on** — flipped in v0.19.2.

## Risk

- **Label correctness across timezones / midnight** — "this morning" vs "yesterday" must use the *local*
  calendar (same engine as A). Mitigated by sharing A's tz-aware helpers + the calendar-day flag; tested at
  the midnight boundary.
- **Verbosity / token cost** — a label per candidate adds a little context. Mitigated by terse labels + the
  existing `RETRIEVAL_K`/slice caps; net negligible.
- **Ordering interactions** — presenting chronologically while selecting by relevance must not drop the
  most-relevant items; selection stays score-based, only the *display order* of the selected set changes.

## Acceptance criteria

- [ ] Every recalled candidate (l2/l3/diary) renders a TS-computed relative-time label and the selected set
      is presented in chronological order, under `LUNA_RECALL_TIME_LABELS`.
- [ ] Labels are correct across the local-midnight boundary and switch to absolute dates past the horizon.
- [ ] Recall *selection* is unchanged (same items chosen); flag-off output is byte-identical to today.
- [ ] `tsc` clean + `bun test` green; ships off.

## Notes for v0.19.2 (don't foreclose)

- The relative-label engine is now the shared truth for "when" — C's felt-absence prose can reference the
  same labels so the subjective layer and the factual layer never contradict.
- Keep label rendering a pure function so a future bi-temporal model (D) can swap in `valid_at` vs
  `created_at` without touching the call sites.
