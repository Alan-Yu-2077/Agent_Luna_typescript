# v0.8.2 — `is_final` promise contract + intent-without-act guard

> **Status: PLANNED.** Initiative: Action Integrity Rails (Order 4, version 3/5). Risk: **Medium**.
> Depends: v0.8.0 (detector), v0.8.1 (contract language). Flag: `LUNA_INTEGRITY_GUARD`
> (default off this version, on at v0.9.0).

## Goal

Mechanical enforcement at the output boundary — the layer that *catches* what the L1 contract only
*asks for*. Generalizes the proven v0.6.2 empty-reply guard from "said nothing" to "promised but
didn't act," with the same one-corrective-retry-then-degrade discipline.

## What ships

- **Generalize the finalize guard** in
  [`runTurn.ts`](../../../packages/server/src/turn/runTurn.ts) (today: the
  `messageTexts.length === 0 …` block at runTurn.ts:297 with `silentRetried`). Replace the single
  boolean with a small retry-reason mechanism:
  - `state.correctionUsed: Set<'empty' | 'promise'>` (bounded — each reason corrects at most once,
    so the existing one-retry guarantee generalizes and can never loop).
  - **Empty reply** (existing): no `message` delivered → `SILENT_TURN_DIRECTIVE` (unchanged).
  - **Broken `is_final` promise** (new, structural): last delivered message had `is_final:false`
    but the turn ended (`finishReason === 'end_turn'`, no pending tool calls). Inject a user-role
    stage direction: "(你说了还有下文却停下了。继续——要么调用工具，要么把话说完并标 is_final。)"
    Re-enter `build_request`. Zero false positives (purely structural).
  - **Intent without act** (new, heuristic): `detectDefection` (v0.8.0) fires on
    **`message_intent`** — a *delivered message text* promised an act and no non-`message` tool
    fired → inject a **double-exit** stage direction: "(如果你打算行动，现在就调用工具——调用就是
    行动本身；如果不打算，就重新说，别用将来时承诺。)" The double exit is deliberate because the
    dictionary is fuzzy: both branches are valid, so a false positive costs one gentle re-prompt,
    never a wrong block. **`thinking_intent` matches NEVER drive a retry** (2026-06-13 audit
    amendment): our thinking is summarized (`display:'summarized'`), so dictionary hits on it are
    low-confidence — they stay audit-only counts in the `decision` traces.
  - All corrective directives are **user-role** (Python v0.27.1 hoisting lesson — never system),
    reuse the existing re-enter-`build_request` path, and emit a `decision` trace
    (`surface:'integrity_guard', decision:'corrected'|'degraded'`) so corrections are countable.
- **Degrade path**: a reason already in `correctionUsed` → no second retry; finalize as-is and
  write a `decision` trace (`decision:'degraded'`). Mirrors the v0.6.2 `empty_turn` degraded
  fallback exactly.
- Detection stays the **pure function** from v0.8.0; this version only wires it into the corrective
  control flow. No new LLM calls on the hot path (the retry is the model's normal next round, not
  an extra judge).

## Tests

- `is_final:false`-then-end → exactly one corrective retry (user-role directive present in the
  re-sent messages); model's retry delivering `is_final:true` closes cleanly; a second
  `is_final:false`-then-end → degraded, one `decision` trace, no loop.
- Intent-without-act (scripted delivered message "我去查一下" + no tool) → one double-exit retry;
  retry that fires the tool → clean; retry that rephrases without promising → clean (both exits
  accepted).
- False-positive safety: message promises + tool fired same turn → no retry. Promise **only in
  summarized thinking** → no retry (audit-only kind), asserted explicitly.
- Bounded: `correctionUsed` guarantees ≤1 retry per reason; a turn hitting both empty and promise
  reasons retries at most once each, never unbounded (assert request count).
- Override-not-depend: with `LUNA_INTEGRITY_GUARD=0`, behavior is exactly v0.8.1 (only the
  empty-reply guard from v0.6.2 remains).
- Full suite green; the v0.6.2 empty-reply tests still pass through the generalized mechanism.

## What this version explicitly does NOT include

No recall tool (v0.8.3). No model-judgment gate — the "intent" detection is a dictionary +
structural check, not an LLM call. The intent dictionary is a starter set; v0.9.0 tunes it against
the recorded false-positive rate.

## Risk

**Medium** — the only place in the initiative with false-positive exposure (the intent
dictionary). Mitigations: (1) the structural `is_final` check carries the high-confidence load and
has zero false positives; (2) the heuristic branch uses a *double-exit* directive so a false
positive is a soft re-prompt, not a wrong correction; (3) bounded to one retry; (4) every
correction traces, so v0.9.0 can measure the false-positive rate and trim the dictionary. The
generalized guard must preserve the exact v0.6.2 empty-reply behavior — regression-tested.

## Acceptance criteria

- [ ] `is_final:false`-then-end is mechanically impossible to ship un-retried (test).
- [ ] Intent-without-act triggers exactly one double-exit retry; both exits accepted; bounded.
- [ ] All corrections/degrades emit `decision` traces.
- [ ] v0.6.2 empty-reply guarantee preserved through the generalized mechanism (its tests green).
- [ ] Flag off → v0.8.1 behavior exactly.
- [ ] Real-LLM smoke: a forced "我去查" turn self-corrects within one retry.

## Notes for v0.8.3 (don't foreclose)

The retry-reason set (`'empty' | 'promise' | 'intent'`) should accept a `'no_recall'` member
without restructuring, in case v0.8.3 wants a gentle "you seemed to need a memory you didn't
recall" nudge. Keep `correctionUsed` a `Set<string>`.
