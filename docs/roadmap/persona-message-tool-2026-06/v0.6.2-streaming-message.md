# v0.6.2 — Streaming Message Text + Empty-Reply Guard

> The bubble streams: `input_json_delta` chunks become `tool.progress{tool_name:'message'}`
> text deltas, the dev chat page renders them live, and a turn can no longer end silent in
> message mode. Master: [`README.md`](README.md).

## Design

**Provider (`anthropic.ts`):** new event `{ kind: 'tool_input_delta', id, name, partial_json }`
— emitted from `content_block_delta/input_json_delta`, keyed to the open tool block from
`content_block_start` (track current block id/name by index). MockProvider gains scripted
support. All other providers/paths unaffected (event ignored where not consumed).

**Incremental text extraction (`packages/server/src/turn/jsonTextStream.ts`):**

A small state machine that, fed successive `partial_json` chunks, emits the unescaped string
value of the top-level `"text"` field as it grows. Spike-verified chunk shapes it must handle:
`["", "{\"text\": \"小猫", "第一次", "看见雪，伸", ..., "\"", ", \"is_final", "\": false}"]` —
chunks split mid-string, mid-escape, mid-key. States: seeking-key → in-string (with escape
carry) → done. ~60 LOC, exhaustively unit-tested (the only genuinely fiddly code in this
initiative — it gets the test density to match).

**runTurn `open_stream` (message mode):**

- `tool_input_delta` for a block named `message` → feed extractor → emit
  `tool.progress { call_id, tool_name: 'message', data: { text_delta } }` per new fragment.
  (`tool.progress` already exists in the protocol — the **payload shape for message is fixed
  here** and recorded for Initiative 6.)
- First text fragment also drives `firstTokenMs` (message mode keeps latency observability).
- Dispatch later validates the **complete** input as usual — streaming preview and validated
  delivery are separate tiers; a `validation_failed` after streamed preview yields
  `tool.finished{err}` and the frontend discards the preview bubble (documented frontend
  contract; dev chat implements it).

**Empty-reply guard (Python v0.47.12 lesson, message mode only):**

- `finalize` checks: zero successful `message` dispatches this turn?
  1. One corrective retry: append a **user-role** stage direction ("你刚才没有说出任何话。
     用 message 工具回应——调用工具就是说话本身。"), re-enter `open_stream`. (User-role, not
     system — Python's v0.27.1 hoisting lesson. Bounded: one retry, reuses the existing
     iteration cap.)
  2. Still silent → fallback: if stray top-level text exists, deliver it as a synthetic
     message envelope (flagged `degraded: true` in trace); else emit `turn.result` with empty
     text + a `node` trace marking `empty_turn` (observable, countable).

**Dev chat page:** message mode rendering — bubble per `message` call (created on first
`tool.progress` delta, finalized/discarded on `tool.finished`), `delay_ms` pacing between
finalized segment reveals, expression key shown as a small chip (no Live2D here — just
visibility). Flag-off rendering unchanged.

## Tests

- jsonTextStream: chunk splits mid-string / mid-escape (`\\"`, `\\n`, unicode) / mid-key;
  text field absent; text field not first key; pathological single-char chunks.
- runTurn (MockProvider): scripted deltas → ordered `tool.progress` text_deltas; preview then
  `validation_failed` → finished err after progress events (frontend-discard contract).
- Empty-reply guard: scripted thinking-only turn → exactly one corrective retry → scripted
  message on retry succeeds; double-silent → degraded fallback path + `empty_turn` trace.
- Flag off: suite green unchanged.
