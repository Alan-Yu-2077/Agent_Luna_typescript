## Design

**Three persona layers (TS mapping of Python's `soul.py`):**

1. **Base persona file** — `packages/server/persona/default.md`, ported/adapted from Python's
   `persona.runtime.default.md` (105 lines, sectioned markdown). Override with
   `LUNA_PERSONA_PATH` env (absolute path). Single user → no active-persona selector file
   (Python layer 2 collapses into the env var).
2. **Core memory prose** — already shipped (v0.4.2 `renderCoreBlock`): `self_state` +
   `relationship_status`, updated by `remember(update_self)` and the dream's `persona_update`
   step. This is the *living* layer the static file can't provide. No new code — the assembly
   order just becomes explicit.
3. **Scene block** — boot framing ported from Python's `build_persona_scene_message`: first
   turn after process boot gets the "newly awakened" framing, later turns the "continuing"
   framing. Implemented as a function of `(bootTurnIndex)` — module-level counter, **not**
   persisted (a restart genuinely is a fresh wake for the lostmemory persona; this is the
   Python semantics and it is correct for this persona).

**System prompt assembly order (one function, `buildSystemPrompt` extended):**

```
[ stable core block, cache_control: ephemeral ]
  1. base directives (existing placeholder)
  2. persona file text          (layer 1)
  3. embodiment paragraph        (ported from soul.py, ~4 lines)
  4. humanity rules block        (caps 140/4/55 stated as prose — guidance tier;
                                  schema enforcement arrives v0.6.1)
  5. core memory block           (layer 2 — existing renderCoreBlock output)
[ message-level, per-turn ]
  6. scene block — injected as part of the FIRST user turn after boot, not system
     (keeps the system core byte-stable across the boot transition)
  7. recall block (existing, unchanged)
```

Order rationale: 1–5 change rarely (persona file edit, core-memory update) → one cache
breakpoint after 5, exactly as today. The scene block is turn-varying, so it lives at message
level like the recall block — **this diverges from Python**, which put scene in system and ate
a cache break on turn 0→1; recorded as part of amendment A5's cache discipline.

**Persona loader (`packages/server/src/persona/loader.ts`):**

- `loadPersona(): { text: string, path: string }` — `statSync` mtime check against module
  cache; re-read only when mtime differs (amendment A5). Missing file → built-in minimal
  fallback string + one boot warning (server must not crash on a deleted persona file).
- `resetPersonaCache()` test seam.

**Humanity rules (`packages/server/src/persona/humanity.ts`):**

- Constants: `MAX_CHARS = 140`, `MAX_SENTENCES = 4`, `MAX_CLAUSE_CHARS = 55` (ported from
  `humanity_rules.json`; TS code is the config — no JSON file, 减负).
- `splitSentences(text)` — ported CJK regex `[。！？!?]+|\n+`; `splitClauses(text)` —
  `[，,；;：:]`. Pure functions, exported for v0.6.1's schema refinement and v0.7.0's
  segmentation.
- `renderHumanityBlock()` — the prose hard-limits paragraph for the system core.

## Tests

- mtime cache: same bytes → same object identity (no re-read); touch file with new content →
  reloaded; missing file → fallback, no throw.
- `buildSystemPrompt` byte-identical across consecutive turns with no persona/core change
  (extends the existing MockProvider snapshot test to include persona blocks).
- Persona file edit between turns → system prompt changes exactly once, then stable again.
- Scene block: present in first user turn after boot, different framing on later turns,
  **absent from system blocks** (cache invariant).
- `splitSentences`/`splitClauses`: CJK + ASCII + mixed punctuation + newline cases (port the
  shapes from Python's auditor tests).
