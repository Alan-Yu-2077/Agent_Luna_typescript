## Design

**Protocol (`packages/protocol/src/message.ts`):**

```ts
export const ExpressionKey = z.enum([
  'curious_attention', 'gentle_concern', 'open_reengagement', 'playful_brightness',
  'focused_engagement', 'steady_presence', 'soft_warmth', 'listening_attention',
  'alert_surprise', 'bright_delight', 'amused_smirk', 'shy_softness',
  'awkward_lightness', 'guarded_distance', 'annoyed_resistance',
]); // Python ALLOWED_AFFECTS, 15 keys, verbatim

export const VoiceParams = z.object({
  provider: z.string().optional(),
  voice: z.string().optional(),
}); // opaque passthrough, backend computes nothing (LD #9)

export const MessageInput = z.object({
  text: z.string().min(1).max(140),
  expression: ExpressionKey.optional(),
  emotion: z.number().min(0).max(1).optional(),
  voice_params: VoiceParams.optional(),
  is_final: z.boolean(),
}).superRefine(/* ≤4 sentences, ≤55-char clause via humanity.ts splitters */);
```

- **Flat root object** (v0.5.2 rule) — `superRefine` carries the sentence/clause caps so the
  wire schema stays a plain object with `properties`. Regression test already enforces shape.
- **`sentences` is NOT a model field** (amendment A1) — derived server-side at dispatch.
- `emotion` is normalized [0,1]; the frontend maps to Live2D's working [0.2, 0.95] intensity
  band (Initiative 6 concern, documented in the field description now).

**Tool (`packages/server/src/tools/builtin/message.ts`):**

- `concurrency: 'session-serial'` (bubbles must arrive in order), `timeoutMs` small (the
  execute is pure — validation already happened; it derives segments and emits).
- `execute`: split `text` via `splitSentences` → segments with `delay_ms = clamp(len*28,
  120, 900)` (amendment A2 — metadata, no sleeps) → yield `ok` with the full delivery
  envelope `{ text, segments: [{index, text, delay_ms}], expression?, emotion?,
  voice_params?, is_final }`. The **`tool.finished` event is the delivery contract** the
  frontend (dev chat now, Initiative 6 later) consumes.
- `summarize`: first 30 chars of text.

**Registry mount + mode switch:**

- `LUNA_MESSAGE_TOOL=1` → registry includes `message`; system core gains the tool-only
  directive block ("you speak only by calling message; never top-level text"); `runTurn`
  ignores stray top-level `text_delta` for the wire (still recorded in history + trace —
  observability of leaks, which the A/B comparison needs).
- Flag off (default) → registry without `message`, no directive block, `reply.token` path
  byte-identical to v0.5.2. **Both modes share one runTurn** — the divergence is registry
  content + system block + a final-text selection rule in `finalize` (message mode: turn text
  = concatenated message texts, for L2/`turn.result` compatibility).

**Turn-loop semantics (message mode):**

- Model calls `message` (possibly several per turn, interleaved with other tools); each
  dispatch emits `tool.started`/`tool.finished` like any tool — *everything-as-tool means no
  special case in the dispatcher* (LD #9's forcing-function point).
- `turn.result` still emitted at finalize with the concatenated text (protocol unchanged —
  existing consumers keep working in both modes).
- Humanity violation → `validation_failed` (recoverable) → model re-emits a shorter message.
  This is the entire enforcement path; no truncation code exists.
