# v0.20.1 — Secret-blocklist hardening

> **Status: PLANNED.** Initiative: Deep-Audit Remediation (Order 13, version 2/10). Risk: **Medium**.
> Depends: nothing (independent of v0.20.0). Flag: none.

## Goal
Close the two confirmed credential-exposure paths that survive the deny-regex: (1) the directory-secret
blocklist is bypassable via shell variable indirection (`cat $HOME/.aws/credentials`), and (2) `grep`'s
JS fallback reads symlinked-in files without re-checking the secret blocklist. Both surface real secret
contents (AWS/GCP/Docker creds, `~/.ssh` keys) into model context within the single-user trust model.

## What ships
**Block `$VAR`/backtick path indirection (`tools/builtin/shell.ts:60-67` `blockedPathInCommand`).**
- The static path-token scanner cannot see through `$HOME`/`${HOME}`/backtick expansion. Treat a path
  token (or command) containing `$`, `` ` ``, or `${` **alongside a sensitive tail** as a refuse signal:
  block any token whose stripped tail path is a directory-secret form (`…/.aws/…`, `…/.ssh/…`,
  `…/.config/gcloud/…`, `…/.docker/…`, `…/.password-store/…`, `…/Library/Keychains/…`) regardless of
  the resolved absolute prefix. This catches `$HOME/.aws/credentials`, `${HOME}/.ssh/id_ed25519`, etc.
- Implement as an extra check in `blockedPathInCommand` (or a sibling `tailSecretInCommand`) that runs
  on the raw token before/besides `resolveInWorkspace`, so it fires even when the resolved path lands
  outside the real home.

**Per-file secret gate in grep's JS fallback (`tools/builtin/grep.ts` jsRunner ~`:126`).**
- Before reading each walked file with `Bun.file(ent.abs).text()`, call `resolveInWorkspace(ent.abs,
  'read')` (mirroring `read_file.ts:54,62`) and skip files it blocks. This closes the symlink-to-secret
  read because `canonicalize()` realpaths the symlink and the secret blocklist then rejects it. (ripgrep
  path is already safe — no `--follow`.)

**Don't emit escaping symlinks from the walker (`tools/fsScan.ts:104-119`).**
- `walk` already skips *descent* into symlinked dirs (`:117`); also stop *emitting* symlinked files
  whose realpath escapes the search root (or add a `followSymlinks:false`/`excludeSymlinks` option the
  security-sensitive callers pass). Defense-in-depth so the gate above is not the only layer.

## Tests
- `shell.test.ts`: the `$HOME`/`${HOME}` forms now assert DENY — `cat $HOME/.aws/credentials`,
  `cat ${HOME}/.ssh/id_ed25519`, `cat $HOME/.config/gcloud/credentials.db`,
  `cat $HOME/.docker/config.json`. Keep the literal-tilde case (`cat ~/.aws/credentials`) green.
- New grep symlink regression: place a symlink `notes.txt → ~/.ssh/id_rsa` (or a temp secret file)
  inside a searched tree, run `grep` forced onto the JS runner (rg absent), assert the secret contents
  do **not** appear in hits and the file is skipped.
- New `fsScan.test.ts` (also satisfies part of v0.20.9's test-debt): symlinked dir not descended;
  symlinked file escaping the root is not emitted; in-root symlinks still work.

## What this version explicitly does NOT include
- The command-injection / deny-regex broadening — **v0.20.0**.
- A general filesystem sandbox / chroot — out of scope (LD #10 owner decision: blocklist, not jail).
- The non-secret directory-traversal "exposure" of ordinary files — not a defect under the no-root-jail
  model; only secret-tier paths are gated.

## Risk
- **Medium.** The token-tail matcher must not over-block legitimate paths that merely contain `.config`
  or `.docker` as a substring of an unrelated dir name — anchor on path-segment boundaries and require
  the secret-segment to be a real path component. Mitigation: reuse the segment-matching logic already
  in `fsScan`/`workspace` rather than a naive `includes`.
- Per-file `resolveInWorkspace` in the JS grep loop adds a stat per file; bounded by the existing
  50000-entry walk cap and only on the rg-absent fallback path, so negligible.

## Acceptance criteria
- [ ] All four `$HOME`/`${HOME}` directory-secret reads assert DENY (regression red→green).
- [ ] grep's JS fallback cannot surface a symlinked-in secret file's contents (regression red→green).
- [ ] `fsScan.test.ts` exists and covers symlink emit/descent behavior.
- [ ] No false-positive blocks on ordinary paths containing secret-like substrings; full suite green.
