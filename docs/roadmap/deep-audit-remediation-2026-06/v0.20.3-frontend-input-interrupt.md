# v0.20.3 — Frontend input & interrupt

> **Status: PLANNED.** Initiative: Deep-Audit Remediation (Order 13, version 4/10). Risk: **Low–Medium**.
> Depends: nothing. Flag: none (frontend behavior fixes).

## Goal
Fix the three frontend defects a real user hits daily: (1) pressing Enter to **commit a Chinese pinyin
candidate** also fires `send()` — broken on essentially every multi-character message in this 中文-first
app; (2) **barge-in is implemented but never wired**, so a new message cannot interrupt Luna's queued
speech; (3) in the documented text-mode escape hatch, consecutive replies **merge into one growing
bubble**. All three are presentation/UX, none touch the wire contract or the default message path's
server logic.

## What ships
**IME-safe Enter (`packages/web/src/app.ts:132-134`).**
- Guard the send: `if (e.key === 'Enter' && !e.isComposing && e.keyCode !== 229) send()`.
  (`isComposing` covers modern browsers; `keyCode===229` is the legacy WebView belt-and-suspenders.)

**Wire barge-in (`audio/webAudioSink.ts:44-48` `stop()` → make it reachable).**
- Call `audio.stop()` at the start of a new user turn: in `app.ts` `send()` (`:124-130`) and/or when
  `controller` sees a new `turn.started` / `chat.send` dispatch. Thread it through the controller deps
  (`controller.ts`) so it stays unit-testable. `stop()` already clears the SerialQueue tail
  (`serialQueue.ts:28-31`), stops the current source, and releases the mouth — it just has no caller.
- Make the in-flight utterance abortable (`audio/audioPlayer.ts:30-42`, `audio/ttsClient.ts:11`): give
  `WebAudioSink` a per-utterance `AbortController`, pass `signal` to the TTS `fetch`, and re-check an
  `aborted` flag (or source identity) **after** `decodeAudioData` resolves before `src.start()` — so a
  `stop()` during decode does not play the now-unwanted clip. (Closes web-audio-2, latent today but live
  the moment barge-in is wired.)
- **Don't let a barge-in abort trip the TTS failure latch.** An `AbortError` from `stop()` cancelling the
  TTS `fetch`/`decode` must be told apart from a real TTS failure: check `err.name === 'AbortError'` /
  `signal.aborted` and **skip** the `fails++` disable-latch increment (`webAudioSink.ts:68-71`), or
  repeated barge-ins would spuriously mute TTS for the whole session. (The latch's *self-heal* is v0.20.8;
  this is the don't-count-intentional-aborts half, and it ships here with the barge-in wiring.)

**Finalize the text-mode reply bubble (`controller.ts:43-49,101-114`).**
- On `turn.result`, if `textStreaming` was true, call `deps.view.finalize(TEXT_BUBBLE, e.text)` before
  resetting `textStreaming=false`. `finalize` stamps + deletes the map entry (`cuteBubbleView.ts:90-100`),
  so the next turn's first `reply.token` opens a fresh bubble instead of re-using the no-op-on-existing-id
  one (`cuteBubbleView.ts:73`). Only reachable under `LUNA_MESSAGE_TOOL=0` (legacy escape hatch), but a
  one-line correctness fix.

## Tests
- `controller.test.ts`: a multi-turn **text-mode** sequence (two `reply.token`+`turn.result` cycles)
  asserts the second reply opens a **new** bubble (the first was finalized + stamped), not appended to
  the first.
- `controller.test.ts`: on a new `send()`/turn start while audio is "speaking", `deps.audio.stop()` is
  invoked (mock asserts the call) — the barge-in wiring.
- `audioPlayer`/`webAudioSink` unit: a `stop()` issued during an in-flight decode does not call
  `src.start()`/`onStart` for that utterance (abort re-check).
- IME guard: a keydown with `isComposing=true` (or `keyCode===229`) does **not** call `send()`; a normal
  Enter does. (Add a small `app`-level or extracted-handler test, since no `app.test.ts` exists today.)

## What this version explicitly does NOT include
- Server-side text-mode path changes — the `LUNA_MESSAGE_TOOL=0` path is legacy/deprecated
  (`runTurn.ts:293-297`); this only fixes the client rendering, not the server emission.
- The keepalive ping / reconnect backoff / `warmUpTts` timeout — **v0.20.8** (resilience bucket).
- The TTS failure-latch self-heal — **v0.20.8**.

## Risk
- **Low–Medium.** Barge-in wiring changes audio timing; mitigation: route through controller deps and
  unit-test the `stop()` invocation + the decode-abort re-check so the "ghost voice" path is covered
  before it can occur. The IME guard is standard and low-risk. Verify in the real browser preview that a
  Chinese message commits cleanly and that sending mid-speech cuts the old audio.

## Acceptance criteria
- [ ] Pressing Enter to select a pinyin candidate does **not** send (manual + unit).
- [ ] Sending a new message interrupts Luna's currently-playing/queued speech (`audio.stop()` fires;
      no stale backlog plays).
- [ ] A `stop()` during decode never plays that utterance.
- [ ] Two consecutive text-mode replies render as two separate, stamped bubbles.
- [ ] Full web suite green; `tsc` clean.

## Notes for v0.20.8 (don't foreclose)
- The per-utterance `AbortController` added here is the seam the v0.20.8 TTS latch self-heal and the
  keepalive work reuse; keep the abort plumbing in `WebAudioSink` so resilience changes don't reshape it.
