# v0.20.4 — Temporal correctness

> **Status: PLANNED.** Initiative: Deep-Audit Remediation (Order 13, version 5/10). Risk: **Low**.
> Depends: nothing. Flag: none (fixes within the already-shipped `LUNA_TIME_AWARE` feature).

## Goal
Fix two confirmed defects in the Initiative-12 time layer: `formatGap` renders impossible labels like
`"1h 60m"` / `"23h 60m"` whenever the within-hour minutes round up to 60 — a narrow window just below
each hour mark (e.g. 1h59m45s), *not* the exact boundary (3600s renders `"1h"` cleanly) — feeding them
to the model and violating the module's own "hand Claude correct labels, never let it compute durations"
contract; and an invalid `LUNA_TZ` **bricks every turn** before the LLM is ever called.

## What ships
**Carry the minute overflow (`turn/temporalContext.ts:136-140`).**
- In the sub-24h branch, after `m = Math.round((s%3600)/60)`, carry on overflow:
  `if (m === 60) { h += 1; m = 0; }`, and if `h >= 24` fall through to the days branch (so `86399` →
  `"1 day"`, not `"23h 60m"` or `"24h"`). A plain clamp-to-59 would give `"23h 59m"` — use the carry +
  fallthrough for the correct label.
- The **sub-hour minutes branch** has the same rounding footgun: `Math.round(s/60)` yields `60` for
  `s ≥ 3570` (e.g. `3599 → "60m"`). Apply the same carry there (`m === 60 → "1h"`) so **no** branch can
  emit a 60-minute label.

**Validate `LUNA_TZ` once (`turn/temporalContext.ts:76-78` `resolveTz`).**
- Probe the zone: `try { new Intl.DateTimeFormat('en-US',{timeZone:tz}); return tz } catch { console.warn(…); return hostZone ?? 'UTC' }`.
  A typo (`Asia/Shanghi`, `PST`) degrades to the host zone instead of throwing. This also protects the
  same-root callers `proactiveTurn.ts:50` (`feltAbsenceFor`) and `recall.ts:249` (`relativeLabel`).
- Belt-and-suspenders: wrap the `buildTimeBlock` call in `runTurn.ts:238-246` in a try/catch that omits
  the time block on failure — the time layer is flag-gated and non-essential, so a temporal failure must
  **degrade, never fail the turn**.

## Tests
- `temporalContext.test.ts`: pin the **carried** cases with correct values — `formatGap(7170)` (1h59m30s,
  minutes round to 60) must render `"2h"` (**not** `"1h 60m"`, and **not** the audit's mistaken `"1h 1m"`);
  `formatGap(86399)` renders a day-level label (**not** `"23h 60m"`); the sub-hour `formatGap(3599)`
  renders `"1h"` (**not** `"60m"`); and `formatGap(3600)` is `"1h"` (the clean boundary was never broken).
  Decide the exact carried rendering (`"2h"` vs `"2h 0m"`) and pin it.
- New: `resolveTz` with `LUNA_TZ='Asia/Shanghi'` returns a valid fallback (not a throw); `buildTimeBlock`
  under a bad `LUNA_TZ` returns a degraded block (or omits it) and a `runTurn` with a bad `LUNA_TZ`
  reaches the LLM (assert `provider.requests.length > 0`, `finishReason !== 'error'`) — the brick is gone.

## What this version explicitly does NOT include
- Any change to time *content* / subjective-time wording (Initiative 12 design stands) — only the
  arithmetic correctness and the config-footgun guard.
- Recall time-label changes — that's the v0.20.5 recall version's neighbor concern (B-layer rendering
  is correct; only the `relativeLabel` tz-robustness is shared and handled here via `resolveTz`).

## Risk
- **Low.** Pure-function fix + a defensive guard. Mitigation: pin the exact carried-label rendering in a
  test so the choice (`"1 day"` vs `"24h"`) is explicit and stable; keep the degrade-path silent except
  for one `console.warn`.

## Acceptance criteria
- [ ] `formatGap` never emits a 60-minute label (`"Nh 60m"` **or** a bare `"60m"`) for any second value
      in `[0, 86400)` (enumerated or boundary-sampled in test).
- [ ] A bad `LUNA_TZ` no longer fails turns — the LLM is reached; a `console.warn` fires once.
- [ ] Time block still byte-identical for valid inputs (cache invariant: the block rides the uncached
      tail, unchanged).
- [ ] Full suite green; `tsc` clean.

## Notes for v0.20.5 (don't foreclose)
- `relativeLabel` (reused by recall's B-layer) now resolves tz safely via the hardened `resolveTz`;
  v0.20.5's recall changes can assume a valid zone.
