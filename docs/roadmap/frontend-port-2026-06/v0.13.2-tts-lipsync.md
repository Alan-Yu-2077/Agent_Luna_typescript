# v0.13.2 — Voice + lip-sync (the real `AudioSink`)

> **Status: PLANNED.** Initiative: Frontend Port — the Body (Order 6, version 4/5).
> Risk: **Medium** (autoplay policy, audio/visual sync, sidecar dependency). Depends: v0.13.1
> (live model + `faceVm.setMouthOpen`). Flag: `LUNA_TTS=0` disables voice → text-only, model still
> animates (silent).

## Goal

Give Luna her voice: a real `AudioSink` that synthesizes each reply via the **GPT-SoVITS** sidecar
(reused from Python), plays it back segment-buffered, fires the **on-audio-start** Live2D commands,
and drives **lip-sync** from the audio's RMS into `faceVm.setMouthOpen`. Port the behavior of Python's
`speech-controller.js` + `audio-stream-player.js` + `lip-sync.js` into TS behind the sink — not a copy.

## What ships

**Decision to settle first (Open Question):** call the GPT-SoVITS sidecar **directly from the browser**
vs **proxy through the TS server**. Recommended: a thin TS-server proxy route (one origin, hides the
sidecar URL, mirrors Python's `Agent/app/server.py` proxy) — but that adds a server route. Record the
choice in the DEVELOPMENT entry.

**New files (`packages/web/src/audio/`):**
- `ttsClient.ts` — POST reply text/segments to the TTS endpoint (proxy route or sidecar), receive
  audio; `voice_params` from the `MessageDelivery` envelope
  ([`packages/protocol/src/message.ts`](../../../packages/protocol/src/message.ts)) select
  provider/voice. Handles the sidecar being down (→ reject cleanly, caller stays silent).
- `audioPlayer.ts` — port of `audio-stream-player.js`: Web Audio `AudioContext`, **segment-buffered**
  playback honoring `segments[].delay_ms`, a `gain` node, and an **analyser** node tapped for RMS.
  Exposes `onStart` (fires when the first buffer actually starts) and `stop()`.
- `lipSync.ts` — port of `lip-sync.js`: read the analyser's RMS per animation frame → a smoothed
  mouth-open 0–1 → `faceVm.setMouthOpen` (via the sink). Stops/zeros on audio end.
- `audioUnlock.ts` — browser **autoplay** handling: the `AudioContext` starts suspended; the first
  user gesture (the first `chat.send` click, or a one-time "🔊 点这里让 Luna 开口" prompt) resumes it.
  Speech requested before unlock is queued, then flushed.
- `webAudioSink.ts` — the real `AudioSink`
  ([`packages/web/src/sinks.ts`](../../../packages/web/src/sinks.ts)): `speak(text, voice, onStart)`
  → `ttsClient` → `audioPlayer.play`; on actual start, invoke `onStart` (the controller's hook) **and**
  set the model to speaking + begin lip-sync; on end, stop lip-sync + return to neutral. `stop()`
  cancels playback + lip-sync.

**Modified files:**
- `packages/web/src/sinks.ts` — `setMouthOpen` (added in v0.13.1) is now actually consumed; no new
  interface growth expected (verify `AudioSink.speak`'s `onStart` is sufficient for the
  `on_audio_start` Live2D-command pattern; extend only if needed, additively).
- `app.ts` — if `LUNA_TTS !== 0`, construct `webAudioSink` (wired to the v0.13.1 `pixiLive2DSink` for
  lip-sync) and pass it to `createController`; else `noopAudioSink`. Install `audioUnlock` on first
  gesture.
- (If proxy path chosen) `packages/server` — a `/tts` route forwarding to the GPT-SoVITS sidecar.
  Keep it minimal + behind the same dev-server handler pattern as `traceViewerHandler`/`devChatHandler`
  in [`packages/server/src/main.ts`](../../../packages/server/src/main.ts).

**Reused (not rebuilt):** the **GPT-SoVITS sidecar** (Python), the v0.13.1 `faceVm`/`pixiLive2DSink`,
the v0.12.0 controller (`audio.speak` is already called on message finalize).

## Tests

- `lipSync`: a scripted RMS series → expected smoothed mouth-open values (pure, fake analyser).
- `audioPlayer`: segment ordering + `delay_ms` honored (mock `AudioContext`/timers; inject the clock).
- `audioUnlock`: speech requested while suspended is queued and flushed after the unlock gesture; none
  dropped.
- `webAudioSink`: `speak` calls `ttsClient`, fires `onStart` once on real start, and drives the
  mock sink's `setState('speaking')` + `setMouthOpen`; `stop()` halts both playback and lip-sync.
- `ttsClient`: sidecar-down → `speak` rejects cleanly and the controller stays silent (no crash,
  text reply already shown).
- **A/V sync + real voice quality is browser-manual** — covered by the smoke below.

## What this version explicitly does NOT include

- **No new chat/visual features** — purely the audio path + lip-sync feeding the existing model.
  Dream overlay, proactive glow, settings, responsive/a11y polish → **v0.13.3**.
- **No TTS caching/prefetch** — synth-on-finalize is fine for now; caching is a later optimization.
- **No multi-voice UI** — voice comes from `voice_params`; a user-facing voice picker is out of scope.

## Risk

**Medium.** (1) **Autoplay policy** — audio won't play before a gesture; mitigated by `audioUnlock` +
queue. (2) **A/V drift** — lip-sync must track real playback, not the request; mitigated by driving
RMS off the live analyser and gating speaking-state on `audioPlayer`'s real `onStart`. (3) **Sidecar
dependency** — if GPT-SoVITS isn't running, voice must fail soft (text still flows); `LUNA_TTS=0` and
the sidecar-down test guard this. (4) **CORS/origin** — the proxy-through-server path avoids it;
recorded as the recommended choice.

## Acceptance criteria

- [ ] TTS path decided (direct vs proxy) + recorded; if proxy, the `/tts` route works.
- [ ] With `LUNA_TTS` on + sidecar running: Luna's replies are spoken; **lip-sync** moves yumi's mouth
      in time with the audio; she enters speaking state on real audio start and returns to neutral on end.
- [ ] First user gesture unlocks audio; a reply that arrived pre-unlock still gets spoken (queued).
- [ ] `LUNA_TTS=0` or sidecar down → silent, text + model animation still work; no crash.
- [ ] `packages/web` (and `packages/server` if proxy) typecheck + tests green.

## Notes for v0.13.3 (don't foreclose)

- `audioPlayer.stop()` + `webAudioSink.stop()` exist — v0.13.3's dream overlay / wake can cut audio
  cleanly when Luna goes to sleep.
- Keep `onStart` as the single "audio actually began" signal — v0.13.3 may also use it to time a
  proactive-message chime or a speaking glow.
- Lip-sync reads from an analyser node — leave the gain/analyser graph intact so a future volume/mute
  toggle (v0.13.3 settings) can sit on the gain node.
