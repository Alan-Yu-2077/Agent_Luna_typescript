## Goal

Make the model stage come alive: render the **yumi** Cubism 4 model with `pixi-live2d-display`, and
implement a real `Live2DSink` that turns each message's `expression`/`emotion` into the avatar's face.
Port the *behavior* of Python's `face-vm.js` layer model (idle ‚Üí state ‚Üí emotion ‚Üí motion) into a
clean TS driver ‚Äî not a line-by-line copy. The model is **draggable** and her position **persists**.
This version is gated so the riskiest piece (WebGL bring-up) can land and be verified before anything
depends on it.

## What ships

**Spike first (record GO/fallback in the DEVELOPMENT entry).** A ~30-line throwaway: load Cubism core
+ pixi + `pixi-live2d-display`, mount yumi into a pixi canvas in `#model-stage`, confirm it renders +
an idle motion plays in the browser. Decide the **integration path**: bundle
`live2dcubismcore.min.js` + `pixi-live2d-display` into `packages/web` vs. `<script>`-tag load (Cubism
core is not an ES module ‚Äî likely a script tag / global, like the Python build). Settle the asset
location for the **yumi** files (copy `models/yumi/` into `packages/web/public/`).

**New files (`packages/web/src/live2d/`):**
- `cubismRuntime.ts` ‚Äî load Cubism core + pixi + `pixi-live2d-display`; create the pixi `Application`;
  guard for **no WebGL** (returns a "degraded" flag so the caller keeps the placeholder).
- `modelDriver.ts` ‚Äî port of `model-driver.js`: load yumi, expose `setParam(id, value)`,
  `playMotion(group, idx)`, `setExpression(name)`; abstracts Cubism specifics. References yumi's
  `.model3.json` parameter/expression IDs.
- `faceVm.ts` ‚Äî port of the `face-vm.js` **layer model** (not the code): a `tick(now, state)` running
  on pixi's ticker (~60fps) that composes layers ‚Äî idle profile ‚Üí current state (neutral / thinking /
  speaking / sleeping) ‚Üí emotion bias ‚Üí active motion ‚Üí param overrides ‚Äî and writes the result via
  `modelDriver.setParam`. (`speakingAmount`/`speechArticulation` inputs are wired but fed 0 until
  v0.13.2 adds lip-sync.)
- `expressionMap.ts` ‚Äî the lookup from the 15 `ExpressionKey` affects
  ([`packages/protocol/src/message.ts`](../../../packages/protocol/src/message.ts)) + `emotion` 0‚Äì1 to
  yumi's concrete expressions/motions/param sets. Built by reading yumi's `.model3.json`. Unknown key
  ‚Üí neutral (never throw).
- `pixiLive2DSink.ts` ‚Äî the real `Live2DSink`
  ([`packages/web/src/sinks.ts`](../../../packages/web/src/sinks.ts)): `setExpression(key, emotion)`
  ‚Üí push an emotion/expression layer into `faceVm`; `clear()` ‚Üí return to idle. **Plus the grown
  interface** (see below).
- `draggable.ts` ‚Äî pointer-drag on the model sprite; persist `{x,y,scale}` to `localStorage`;
  **double-click resets to center**. Bounded so she can't be dragged off-screen.

**Modified files:**
- `packages/web/src/sinks.ts` ‚Äî **grow `Live2DSink`** (additive): add `setState(state: 'neutral' |
  'thinking' | 'speaking' | 'sleeping')` (and a placeholder `setMouthOpen(v: number)` consumed in
  v0.13.2). Update `consoleLive2DSink`/`noopAudioSink` stubs to match so v0.13.0's wiring still
  compiles.
- `packages/web/src/controller.ts` ‚Äî drive the new states: `turn.started` ‚Üí `live2d.setState('thinking')`;
  first message delta / `audio.speak` start ‚Üí `'speaking'`; turn end ‚Üí `'neutral'`; `dream.status
  is_dreaming` ‚Üí `'sleeping'`, wake ‚Üí `'neutral'`. (These are new controller‚Üísink calls; covered by
  controller tests with a spy sink.)
- `app.ts` ‚Äî if `LUNA_LIVE2D !== 0` and WebGL present, construct `pixiLive2DSink` (mount into
  `#model-stage`) and pass it to `createController`; else keep the placeholder + `consoleLive2DSink`.

**Reused (not rebuilt):** `pixi-live2d-display`, `live2dcubismcore`, the **yumi** model assets, and the
v0.12.0 controller event-routing.

## Tests

- `expressionMap`: every one of the 15 `ExpressionKey`s maps to a defined yumi expression/param set;
  an unknown key ‚Üí neutral, no throw. (Pure data test, no WebGL.)
- `faceVm`: given a scripted state sequence + a fake clock, `tick` produces the expected layer
  composition / `setParam` calls (mock `modelDriver`). Deterministic ‚Äî inject `now`, never read the
  real clock.
- `pixiLive2DSink`: `setExpression`/`setState`/`clear` push the right layers into a mock `faceVm`.
- `controller`: with a spy `Live2DSink`, assert the new state transitions fire on
  `turn.started`/message-start/turn-end/`dream.status`/wake (extends the v0.12.0 controller suite).
- `draggable`: a simulated drag updates + persists position; double-click restores center
  (jsdom/pointer events; localStorage mocked).
- **WebGL/render itself is browser-manual** (can't unit-test the GPU): the spike + the manual smoke
  cover it.

## What this version explicitly does NOT include

- **No audio, no lip-sync.** `faceVm`'s `speakingAmount`/`setMouthOpen` are wired but fed 0;
  `audio` is still `noopAudioSink`. Real voice + RMS-driven mouth ‚Üí **v0.13.2**.
- **No dream overlay visuals** beyond `setState('sleeping')` posing the model ‚Üí **v0.13.3**.
- **No emotion-mood indicator UI, no proactive glow** ‚Üí **v0.13.3**.

## Risk

**High** ‚Äî the WebGL/Cubism bring-up is the single biggest unknown in the initiative. Mitigations:
(1) **spike before building**, record the GO decision; (2) **`LUNA_LIVE2D=0` flag** + WebGL-absent
detection ‚Üí the app always falls back to the v0.13.0 placeholder and chat keeps working; (3) keep the
`faceVm` layer logic behind a mockable `modelDriver` so the *behavior* is unit-tested without a GPU;
(4) yumi's exact expression/motion/param IDs are unknown until read from `.model3.json` ‚Äî the spike
surfaces them before `expressionMap` is written. Asset licensing/path: copy yumi into
`packages/web/public/` (same assets the Python build already ships).

## Acceptance criteria

## Notes for v0.13.2 (don't foreclose)

- `faceVm` already accepts `speakingAmount`/`speechArticulation` and the sink already has
  `setMouthOpen` ‚Äî v0.13.2 only needs to feed real RMS values in; no interface change.
- The `'speaking'` state is entered by the controller on message start; v0.13.2's `audio.speak`
  `onStart` callback should align the mouth-open ramp with actual audio start (the `on_audio_start`
  pattern), so keep the speaking-state entry point swappable to the audio callback.
- Keep `modelDriver.playMotion` exposed ‚Äî v0.13.2/v0.13.3 may trigger gesture motions on audio start.
