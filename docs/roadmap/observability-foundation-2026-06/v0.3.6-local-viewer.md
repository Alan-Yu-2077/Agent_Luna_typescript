# v0.3.6 — Local viewer

> **Status: SHIPPED 2026-06-11.** Initiative: observability-foundation (Order 1.5, version 2/2).
> Risk was Low. Depends: v0.3.5. Flag: `LUNA_VIEWER` (default on; `=0` disables → server becomes
> WS-only and `/_trace` 426s, a deliberate divergence from the planned 404). `LUNA_TRACE` default
> flipped to on. See `docs/history/DEVELOPMENT.md` for the v0.3.6 record.

## Goal

A developer can open `http://localhost:8787/_trace` and see the last N turns rendered as a
flat per-turn timeline. No external tooling. No build step. Reads the SQLite trace table
populated in v0.3.5.

After this version, `LUNA_TRACE` defaults to **on** locally — because there is now a viewer
to look at what's stored.

## What ships

### `packages/server/src/trace/viewer.ts` (new)

- Registers three routes on the existing `Bun.serve` instance:
  - `GET /_trace` — serves `viewer/index.html` (static).
  - `GET /_trace/api/turns?limit=50` — returns `{ turns: [...] }` (latest first).
  - `GET /_trace/api/events?turn_id=X` — returns `{ events: [...] }` (ascending
    `started_ms`).
- Read-only. No mutation routes. No auth (localhost only).

### `packages/server/src/trace/viewer/index.html` (new)

- ~150 LOC: HTML + CSS + vanilla JS, single file, no framework.
- **Left pane**: list of recent turns (timestamp, duration_ms, event count).
- **Right pane**: timeline of selected turn — each node transition + tool call as a row with
  a duration bar.
- Auto-refresh every 2s.
- Click a row to expand `payload_json`.

### Conditional mounting

The `/_trace*` routes mount **only** when `LUNA_VIEWER !== '0'`. Default on locally, can be
disabled in any non-localhost binding when we eventually have one. Block in `main.ts`:

```ts
if (Bun.env['LUNA_VIEWER'] !== '0') registerTraceViewer(server);
```

### `LUNA_TRACE` default flip

v0.3.5 ships with `LUNA_TRACE=0` (path proven, off). v0.3.6 ships with `LUNA_TRACE=1`
(default on locally) — the viewer makes traces useful, so default-on is now reasonable.

## Tests

1. **HTML serves**: `GET /_trace` returns HTML with status 200. Assert `<title>` matches.
2. **API turns**: send 3 `chat.send` events with `LUNA_TRACE=1`. `GET /_trace/api/turns`
   returns 3 entries with descending `started_ms`.
3. **API events**: `GET /_trace/api/events?turn_id=<id>` returns the events for that turn
   in ascending `started_ms`; first row is `kind='node'` with `node_from='_'`.
4. **Disable flag**: with `LUNA_VIEWER=0`, `GET /_trace` returns 404.
5. **Non-interference**: existing v0.3 WS round-trip tests stay green with the viewer
   mounted.

## What v0.3.6 explicitly does NOT include

- No charts or aggregations — flat list per turn.
- No filters / search beyond `?turn_id`.
- No authentication (localhost only, by design).
- No frontend framework — vanilla JS, single file.
- No export / share.
- No live tail of in-flight turns (only completed, flushed turns are visible).

If real use shows any of these are needed, add them in a follow-up patch.

## Risk

**Low.** Read-only. Single static page. If the viewer crashes, chat still works.

The one watch point: don't let viewer requests interfere with the WS connection on the same
port. Bun.serve handles HTTP + WS on the same port natively, so this should be free, but
verify in test 5.

## Notes for Initiative 2 (don't foreclose)

- The trace store read API is the same one Initiative 2 memory tools can use for "show me
  when this memory was last referenced." Keep `listTurns`, `getEventsByTurn`, `getTrace`
  stable.
- The viewer's left-list / right-detail layout is a candidate pattern for the v0.12 frontend
  debug overlay if we add one — but don't bake hard dependencies on its current shape.
