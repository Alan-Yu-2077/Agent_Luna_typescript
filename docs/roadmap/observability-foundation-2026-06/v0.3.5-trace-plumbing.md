# v0.3.5 — Trace plumbing

> **Status: PLANNED.** Initiative: observability-foundation (Order 1.5, version 1/2). Risk:
> **Medium** (first `bun:sqlite` touch; establishes WAL + migration pattern v0.4 memory
> reuses). Depends: v0.3.0. Flag: `LUNA_TRACE=1` (default **off** in v0.3.5; default **on**
> after v0.3.6 ships the viewer).

## Goal

Every turn-loop node transition (per the v0.3 StateGraph) and every tool dispatch writes a
typed trace row to SQLite, all linked by a `trace_id` that propagates through the turn. After
this version, every Luna turn is queryable as a structured event tree.

This is also the first `bun:sqlite` touch in the rewrite. The WAL-mode + migration pattern
established here is copied verbatim by v0.4 (memory substrate).

## What ships

### `packages/protocol/src/trace.ts` (new)

Zod schemas for trace events — one discriminator per kind. Same pattern as `events.ts`:

- `NodeTraceEvent = { kind: 'node', trace_id, turn_id, node_from, node_to, t_ms }`
- `ToolTraceEvent = { kind: 'tool', trace_id, turn_id, call_id, tool_name, phase: 'started' | 'progress' | 'finished', t_ms, payload }`
- `OutboundTraceEvent = { kind: 'outbound', trace_id, turn_id, server_event_type, t_ms }`
- `TraceEvent = z.discriminatedUnion('kind', [...])`

Each event also carries `schema_v: z.literal(1)` so the migration anchor is in the data, not
just the DB.

### `packages/server/src/sql.ts` (new, **extracted for v0.4 reuse**)

The shared `bun:sqlite` boilerplate. v0.4 memory substrate imports this unchanged.

- `openDb(path) → Database` — WAL mode, foreign keys on, busy_timeout = 5000ms.
- `migrate(db, migrationsDir)` — reads `PRAGMA user_version`, applies missing `.sql` files in
  order, bumps version. Idempotent.
- `closeDb(db)` — graceful close on shutdown.

### `packages/server/src/trace/` (new directory)

```
trace/
├── README.md       ← documents the instrumentation pattern; v0.4+ reads this
├── store.ts        ← per-turn buffer + flush; uses sql.ts
├── id.ts           ← traceIdFor(session_id, turn_seq) — deterministic
├── instrument.ts   ← trace(event: TraceEvent): void — single emit entry point
└── migrations/
    └── 0001_traces.sql
```

### Schema (v1)

```sql
-- packages/server/src/trace/migrations/0001_traces.sql
CREATE TABLE traces (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  schema_v     INTEGER NOT NULL,
  trace_id     TEXT NOT NULL,
  turn_id      TEXT NOT NULL,
  session_id   TEXT NOT NULL,
  kind         TEXT NOT NULL,           -- 'node' | 'tool' | 'outbound'
  payload_json TEXT NOT NULL,
  started_ms   INTEGER NOT NULL,
  ended_ms     INTEGER                  -- nullable for fire-and-forget events
);
CREATE INDEX idx_traces_turn    ON traces(turn_id, started_ms);
CREATE INDEX idx_traces_session ON traces(session_id, started_ms);
PRAGMA user_version = 1;
PRAGMA journal_mode = WAL;
```

### Instrumentation points (wrap, don't replace)

- **Inbound** (`ws.ts:handleMessage`): on `chat.send`, mint `trace_id =
  traceIdFor(session, turn_seq)`; store in turn state; emit
  `NodeTraceEvent({node_from:'_', node_to:'parse_input'})`.
- **Outbound** (`outbound.ts:outbound`): emit `OutboundTraceEvent` after `ServerEvent.parse`,
  before `ws.send`. Single line addition.
- **StateGraph edges** (v0.3 `runTurn.ts`): each transition calls
  `trace({kind:'node', node_from, node_to, t_ms: Date.now()})`. One call per edge.
- **Dispatcher** (v0.2 `dispatcher.ts`): emit `ToolTraceEvent` at each phase. `'progress'`
  events are batched (every Nth or every 50ms, whichever first) to avoid flood.

### Buffering policy

- `store.ts` buffers `TraceEvent[]` in memory keyed by `turn_id`.
- Flush on `node_to === 'end'` (turn completes) in a single transaction.
- Flush on `SIGTERM` (graceful shutdown).
- Hard cap: 500 events per turn. Overflow drops with a single `trace.overflow` row
  carrying `dropped_count`.

### Flag behavior

`LUNA_TRACE=1` enables write. Default **off** in v0.3.5 — proves the path works without
committing to always-on writes. Flipped to default **on** in v0.3.6 once a viewer can show
what's stored.

### Q4 dependency

The `payload_json` column's truncation policy depends on Open Q #4 (Result compaction
tripwire). Three options:
- (a) Store full tool output (simple, scales poorly)
- (b) Truncate at 4KB with `truncated: true` marker
- (c) Store hash + side-table blob storage

**Resolve Q4 before locking the schema.** If unresolved at implementation time, default
to (b) since it's reversible (can rewrite migration when Q4 resolves).

## Tests

1. **Trace row exists**: send `chat.send`, complete turn, query
   `SELECT COUNT(*) FROM traces WHERE turn_id=?`. Assert ≥ 5 rows (input + ≥2 node
   transitions + outbound + final).
2. **`trace_id` consistency**: all rows for one turn share the same `trace_id`. Reconnect
   mid-turn (mirrors v0.3 test 5) keeps the `trace_id`.
3. **Schema validation**: rows where `kind='tool'` parse against `ToolTraceEvent`. Rows
   where `kind='node'` parse against `NodeTraceEvent`. Etc.
4. **Buffering / overflow**: a turn with 600 events writes 500 rows + one `trace.overflow`
   row with `dropped_count=100`.
5. **Migration idempotency**: fresh DB starts at `user_version=0`; after boot,
   `user_version=1` and the `traces` table exists. Restarting doesn't re-run the migration.
6. **Latency budget**: 100 synthetic turns end-to-end with `LUNA_TRACE=1` complete within
   105% of the same 100 turns with `LUNA_TRACE=0`.

## What v0.3.5 explicitly does NOT include

- Viewer / UI — that's v0.3.6.
- Trace-driven eval (eval framework is a separate future initiative).
- Distributed trace correlation (no W3C `traceparent`). Local SQLite only.
- Sampling — every turn writes when `LUNA_TRACE=1`. Sampling is v0.5+ territory.
- Retention policy — DB grows unbounded for now. Add a `prune.ts` if it bites.

## Risk

- **First `bun:sqlite` touch**: connection pool, WAL mode, migration pattern, busy-timeout
  all introduced here. **Mitigation**: extract into `packages/server/src/sql.ts` so v0.4
  reuses the same code, not a parallel implementation.
- **Latency regression**: per-turn buffer + post-turn flush should keep <5%. If a turn
  legitimately emits hundreds of `tool.progress` events (long-running streaming tool),
  flush at the 500-event boundary mid-turn rather than waiting for completion.
- **Migration bootstrap on a fresh clone**: zero-state DB needs to auto-migrate to v1
  without manual SQL. `migrate()` checks `PRAGMA user_version` and runs missing migrations
  on boot.

## Notes for v0.3.6 (don't foreclose)

- The trace store must expose a read API that's stable for the viewer:
  `listTurns({limit, before_ms})`, `getEventsByTurn(turn_id)`, `getTrace(trace_id)`. Design
  these in v0.3.5 even though they're not consumed yet.
- Keep `payload_json` as raw text in the read API; let the viewer call `JSON.parse` on the
  way out. Symmetric with the outbound boundary's "parse only at the edge" discipline.
- Index `(turn_id, started_ms)` and `(session_id, started_ms)` cover the viewer's intended
  queries; don't add more indexes until measured.
