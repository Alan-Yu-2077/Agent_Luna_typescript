# v0.25.1 — Collapse/expand mode + window↔bottom-bar morph

> **Status: PLANNED.** Initiative: Collapsible companion UI (Order 18, version 2/3). Risk: **Medium**.
> Depends: v0.25.0 (the stack must exist so collapsed replies have somewhere to go). Flag: the collapse
> control itself is the gate; `luna:collapsed` persists the state. **`packages/web` only.**

## Goal

Assemble the collapsed companion **experience**: a control that gracefully morphs the chat window into
a single bottom input bar, persists the mode, hides the window's message log, and — via the
`RouterBubbleView` from v0.25.0 — routes Luna's replies to the beside-model stack instead of the
window. Expanding reverses it. No model motion yet (v0.25.2); the model simply re-centers when its
region resizes.

## What ships

**Modify — `packages/web/src/ui/layout.ts`**: add a collapse/expand toggle control (e.g. a small caret
button on the chat header or a dedicated `.collapse-btn`) and return it in `LayoutRefs` (extend the
type at `layout.ts:8-27`). No structural DOM change beyond the button — the `.chat-input-row`
(`layout.ts:142-154`) is reused as the collapsed bar.

**Modify — `packages/web/src/app.ts`** (state + wiring):
- A `collapsed` client state boolean, initialized from `localStorage['luna:collapsed']`, toggled by the
  new button, applied as a `.collapsed` class on `.luna-app` (mirroring the `.settings-panel.on` /
  `luna:reduce-motion` idiom, `app.ts:30,153`). Persist on change.
- Repoint the `RouterBubbleView`'s `collapsed` getter (from v0.25.0) at this real state (replacing the
  `luna:bubble-stack` test toggle) so collapsing routes replies to the stack + expanding routes them to
  the window.
- **User echo:** in `send()` (`app.ts:136-149`), when collapsed, **suppress** the window echo
  (`view.userMessage`) — the user sees their text in the input as they type — or optionally push a brief
  distinct user bubble to the stack (build-time choice, README open question). The WS `chat.send` is
  unchanged.
- On collapse/expand, dispatch a synthetic `window 'resize'` (or call an exposed re-fit) so the model
  re-centers into the resized `.model-stage` (the actual *glide* is v0.25.2; here it may snap).

**Modify — `packages/web/src/ui/theme.css`** (the morph):
- A `.collapsed` rule set that morphs `.chat-panel` → a single bottom input bar and grows
  `.model-stage` to full width. Because transitioning to/from `position:fixed` is not animatable, use a
  **cross-fade / FLIP-style** approach: e.g. keep the input-row in flow but transition the panel's
  height/opacity while a bottom-anchored bar fades in — or measure-and-animate. Hide `.chat-header`,
  `.chat-log`, `.scroll-pill` in collapsed mode.
- **Reconcile with the mobile breakpoint** (`theme.css:171-175`, which already reflows `.stage` to a
  column at ≤720px) so collapsed + narrow don't fight (e.g. scope collapsed rules to not double-apply
  the column reflow).
- `.reduce-motion` / `@media (prefers-reduced-motion)` overrides: the morph snaps instead of animating.

## Tests (`bun test packages/web/`)

- `app`/layout wiring test (jsdom): toggling collapse adds/removes `.collapsed` on the root and
  persists `luna:collapsed`; the collapse button exists in `LayoutRefs`.
- Router integration: with `collapsed` true, a `finalize` reaches the stack (+ window); with it false,
  only the window — reusing the v0.25.0 router test harness against the real state getter.
- Echo: in collapsed mode `send()` does not push a user bubble into the window log (or pushes to the
  stack per the chosen option); `chat.send` payload is unchanged either way.
- Reload persistence: a stored `luna:collapsed=1` boots collapsed.

## What this version explicitly does NOT include

- **The model glide** (smooth center↔right motion) → v0.25.2. Here the model re-centers via a re-fit
  (may snap).
- Any `packages/server` / `packages/protocol` change.

## Risk

- **Medium — pure-CSS morph of a flex panel into a bottom bar** (`position:fixed` isn't animatable).
  Mitigation: FLIP-style / cross-fade approach; reduce-motion snap as the guaranteed-correct fallback;
  the morph is cosmetic — correctness (replies reach the stack) is independent of the animation.
- **Collapsed + mobile conflict** (`theme.css:171-175`). Mitigation: explicit reconciliation + a test at
  a narrow viewport.
- **Lost replies if the flag repoint is wrong.** Mitigation: the router forwards to the window
  **always** + the stack when collapsed, so a mis-set flag degrades to "also shows in window", never
  "reply vanishes".

## Acceptance criteria

- [ ] The collapse control morphs the window into a bottom input bar and back; state persists across
      reload; instant under reduce-motion.
- [ ] Collapsed: user sends via the bottom bar (`chat.send` unchanged), Luna's replies appear as
      beside-model bubbles and **not** in the window; expanded: replies in the window as today.
- [ ] Collapsed + narrow viewport lays out sanely (no fight with the ≤720px reflow).
- [ ] `packages/server` + `packages/protocol` untouched; `bun test` + `tsc --noEmit` (web) green.

## Notes for v0.25.2 (don't foreclose)

- Keep the on-collapse re-fit call swappable for the glide tween so v0.25.2 replaces the snap with an
  eased motion without touching the collapse wiring.
