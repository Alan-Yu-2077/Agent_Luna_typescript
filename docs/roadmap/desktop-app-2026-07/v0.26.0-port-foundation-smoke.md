# v0.26.0 — Port foundation + Electron rendering smoke test

> **Status: PLANNED.** Initiative: Desktop app (Order 19, version 1/3). Risk: **High** (validates the
> load-bearing assumption: does the whole stack render + drive + connect inside the shell). Depends:
> nothing. Flag: none (a new `packages/desktop`; `packages/web` stays browser-runnable, no regression).

## Goal

De-risk the engine + the plumbing before any packaging work: produce a **static production build** of
`packages/web`, serve it over a **fixed loopback HTTP origin** (so `location.hostname`-derived URLs +
absolute asset paths + `localStorage` all keep working unchanged), and load it in a **plain Electron
window** that proves **yumi renders, drives (breathing/idle/lipsync), and the WS loop connects**. The
server is still hand-started this version — the hard packaging (compile + supervise the server) is
v0.26.1. This isolates the riskiest unknown (does the whole loop work in the shell) with the least
moving parts.

## What ships

**New `packages/desktop/`** (Electron, minimal):
- `main.ts` — Electron main: a small **fixed-port loopback static server** (reuse `dev-server.ts`'s
  logic: serve `dist/` + the `/api/gpt-sovits/` proxy), then a `BrowserWindow` loading
  `http://127.0.0.1:<fixedPort>`. A **plain** window (default chrome) this version — the transparent
  pet window is v0.26.2.
- `preload.ts` — empty/minimal (context-isolation scaffold for later).
- `package.json` — Electron dep + a `dev` script; wired into the workspace.

**Modify `packages/web/package.json`** — add scripts:
- `build`: `bun build ./index.html --production --outdir=dist` (Bun's bundler transpiles
  `src/app.ts` + theme.css; `dev-server.ts:2` confirms the HTML entry).
- `build:assets`: copy `public/live2dcubismcore.min.js` + `public/models/` into `dist/` (the bundler
  does **not** copy `public/`; the absolute-root fetches need them present).

**Modify `packages/web/src/app.ts`** (the one load-bearing URL fix, `app.ts:23-24`):
- Make the WS URL **config-driven** with a safe default: keep the `?ws=` dev override, but default to a
  fixed `ws://127.0.0.1:8787` (not `location.hostname`) so it is correct regardless of how the shell
  serves the page. (Served over the loopback HTTP origin, `location.hostname` already yields
  `127.0.0.1` — this change makes it robust + explicit rather than incidental.)

**No `packages/server` change** this version (it is hand-started, listening on `127.0.0.1:8787`).

## Tests

- `packages/web`: a unit test that the WS-URL resolver returns `ws://127.0.0.1:8787` by default and
  honors a `?ws=`/config override (pure function extracted from the `app.ts:23-24` inline).
- Build smoke (script, not a bun test): `bun run build && bun run build:assets` produces a `dist/` whose
  `index.html` references hashed JS/CSS and that contains `live2dcubismcore.min.js` + `models/yumi/`.
- **Manual go/no-go (the real acceptance):** launch the Electron window against a hand-started server →
  yumi renders in the window, animates (idle/breath), a chat turn round-trips over WS, TTS proxy
  reachable (or degrades cleanly). This is the engine-certainty checkpoint.

## What this version explicitly does NOT include

- **Bundling/spawning the server** (compile sidecars, supervision, app-data paths) → v0.26.1.
- **The transparent pet window** (transparency, always-on-top, click-through, throttling) → v0.26.2.
- **The geo shim** → v0.26.2 (weather falls back to `LUNA_LAT_LON` until then).
- Any Tauri work (Electron is the chosen v1 shell; Tauri is a deferred footprint optimization).

## Risk

- **High — this is the engine/loop validation.** If yumi somehow does not render or the WS loop fails in
  Electron, the whole initiative's shell choice is wrong — which is exactly why this is v0.26.0 and
  minimal. Mitigation: Electron ships the same Chromium the browser build already targets, so the
  adversarial verdict rates rendering risk **low** here; the value is proving the *end-to-end* loop
  (serve → load → WS → render) with nothing else in the way.
- **Asset paths 404 if the origin isn't a real HTTP origin.** Mitigation: serve `dist/` over a fixed
  loopback HTTP port (not `file://`), which keeps every absolute-root fetch working.
- **`localStorage` reset if the port floats.** Mitigation: pin the loopback port.

## Acceptance criteria

- [ ] `bun run build && bun run build:assets` yields a self-contained `dist/` (bundle + Cubism core +
      yumi model).
- [ ] The Electron window loads `dist/` over a fixed loopback origin; **yumi renders + animates**; a
      chat turn round-trips over WS to a hand-started server.
- [ ] The WS URL is config-driven (default `ws://127.0.0.1:8787`); `packages/web` still runs in a
      browser unchanged.
- [ ] `bun test` + `tsc` green.

## Notes for v0.26.1 (don't foreclose)

- Keep the loopback static server + `/api/gpt-sovits` proxy as a **standalone module** so v0.26.1 can
  `bun build --compile` it into a sidecar (or keep it in the Electron main) without rework.
- Keep the WS URL + loopback port read from a **single config object** so v0.26.1 can point them at the
  spawned server + a chosen port with no code churn.
