# v0.14.1 — Cross-platform one-command launcher + setup

> **Status: PLANNED.** Initiative: Open-source packaging (Order 7, version 2/3). Risk: **Medium**
> (cross-platform process orchestration). Depends: v0.14.0. Flag: none.

## Goal

Make "run the whole thing" literally one command on macOS / Linux / Windows: a single `bun run dev`
(and a `bun start`) that brings up `packages/server` (8787) + `packages/web` (5173) + the GPT-SoVITS
sidecar together, with readable multiplexed logs and clean teardown. Plus a `bun run setup` that does
the one-time heavy lifting (pull the Docker image + fetch yumi's voice model/weights) so the first run
is "setup once, then one command forever."

## What ships

**Root `package.json` scripts:**
- `dev` — uses **`concurrently`** (devDep; actively maintained, no Docker/Python/Ruby needed for the
  runner itself) to launch three named processes: `server` (`bun run packages/server/src/main.ts`),
  `web` (`bun packages/web/dev-server.ts`), `tts` (`docker compose -f services/tts/gpt-sovits/docker-compose.yml up`).
  Colored `--names server,web,tts` + `--kill-others-on-fail`.
- `start` — same shape for a non-watch run (or document it as an alias for now).
- `setup` — `bun run services/scripts/setup.ts`: checks Docker is installed, pulls the GPT-SoVITS
  image, fetches the base weights + yumi's voice model into the gitignored `models/` (HF /
  `HF_ENDPOINT` mirror / ModelScope), and prints next steps. Idempotent; clear progress + sizes.
- Keep `dev:server` / `dev:web` as the existing single-service scripts (the launcher composes them).

**New `services/scripts/setup.ts`** — the fetch/bootstrap (model download + Docker pull + health
hint). Honest output: states the ~5 GB download + Docker requirement up front.

**Cross-platform notes baked in:** `concurrently` runs everywhere; document the known Windows caveat
(Node-based runners can't deliver a clean SIGTERM on Windows — Ctrl-C still stops them, the Docker
container is stopped via `docker compose down` in a `predev`/`postdev` or documented step).

## Tests
- `package.json` has `dev` / `start` / `setup`; `dev` invokes `concurrently` with the three commands
  (assert via reading the script string in a small test, or a lint check).
- `setup.ts` pure parts (URL/mirror selection from `HF_ENDPOINT`, idempotent skip-if-present) unit-tested
  with a stubbed fetch/fs.
- **Owner-verified (Docker):** fresh-ish clone → `bun install` → `bun run setup` → `bun run dev` → all
  three up, browser shows Luna, she speaks in yumi's voice; Ctrl-C tears all down.

## What this version explicitly does NOT include
LICENSE / README / `.env.example` / `THIRD_PARTY_LICENSES` / secret-scan → **v0.14.2**. No change to
the TTS translation (v0.14.0) or the agent/web runtime.

## Risk
**Medium.** Concerns: cross-platform shell differences (mitigate by using `concurrently` + plain `bun`/
`docker compose` invocations, no shell-isms); Windows teardown (documented caveat); Docker not
installed (the `tts` process fails — `--kill-others-on-fail` would stop the app, so consider making
`tts` non-fatal / a separate `dev:full` vs `dev:app` so the app still runs sans voice). Decide the
fail-soft behavior so "no Docker" still gives a working (silent) Luna.

## Acceptance criteria
- [ ] `bun run setup` then `bun run dev` (one command) starts server + web + TTS together, cross-platform.
- [ ] Logs are name-prefixed; Ctrl-C stops the Bun services + the Docker sidecar.
- [ ] "No Docker installed" degrades to app-runs-silent (not a hard crash) — fail-soft decided + documented.
- [ ] Unit tests green; owner confirms the full one-command flow with Docker.
