# v0.16.1 — Recompute efficiency (memoize / cap / off-hot-path)

> **Status: PLANNED.** Initiative: Audit remediation (Order 9, version 2/3). Risk: **Low–Medium**.
> Depends: v0.16.0 (CI gate protects the refactors). Flag: `LUNA_RECALL_ASYNC` (P1 only).

## Goal

Kill the per-turn recomputation the efficiency audit found, without changing any output. Three of the
four are pure server-side waste (same bytes, recomputed); one (P1) takes the recall embedding off the
critical path to the first LLM token. This is the slice that makes the project's #1 goal (speed) real,
and it's the **prerequisite** for Initiative 10's deeper window (a ~100-turn window over today's per-turn
recompute would be painful; over a memoized/capped one — plus v0.16.3's clean turns — it's fine).

## What ships

### A1 — memoize the system block on the turn
The system prompt is rebuilt every tool iteration (`runTurn.ts:192`, `open_stream` looped ≤8×), each
costing 6 DB queries + an uncached L1-contract build, to produce a byte-identical block. Memoize it on
`TurnState`, mirroring the existing tool-schema memoization (`runTurn.ts:179`):
- cache `s.systemBlock` once built; reuse across iterations.
- invalidate on a cheap **dirty flag** set when `core_memory` or `l3_facts` change mid-turn (the
  `remember`/`update_self` tools) — so a mid-turn memory edit still re-renders, but only then.
- also add a module-level `let cached` to `renderL1Contract` (`persona/l1Contract.ts:6`) — it's a pure
  constant string; build once.

### A4 — trace retention + bounded viewer query
`traces` grows unbounded (no `DELETE FROM traces`), and `listTurns` aggregates the whole table
(`trace/store.ts:95`) on a 2 s viewer refresh.
- add a **retention window** (drop traces older than N turns/days; the dream cycle is the natural prune
  point — it already runs periodically). Migration-free if it's a `DELETE` on a schedule; if a config
  table is wanted, a versioned migration.
- bound `listTurns` to a recent time range / `LIMIT`, and the viewer to that window.

### A2 — recall: read only what's needed + persist content hashes
`retrieve()` over-fetches L2 (up to 10 000 rows to use 500) and re-sha256-hashes all ~500 candidates
every call (`recall/recall.ts:114,158`), hashes never persisted.
- pass `{ limit: L2_CANDIDATE_LIMIT }` (500) with `ORDER BY t_ms DESC` to `listL2`
  (`sessionStore.ts:114`) so only the needed rows are read.
- store the `content_hash` as a column on `l2_turns` and `l3_facts` (computed once at insert) →
  **versioned migration** + backfill; read it back instead of re-hashing in `retrieve`. Keeps the
  `embeddings_cache` keying identical, just stops recomputing the key.

### P1 — recall query-embed off the TTFT path (flag `LUNA_RECALL_ASYNC`)
`parse_input` `await`s `retrieve()` before `open_stream` (`runTurn.ts:168`→`185`), so a cold-cache turn
does 1–2 embedding round-trips before the first token. Behind `LUNA_RECALL_ASYNC` (default off → on
after measuring):
- either **overlap** the query embedding with the LLM first-byte (kick off `open_stream` and inject
  recall as a follow-up message block only if it returns within a budget), **or**
- give recall a **latency budget** that falls back to lexical-only (the bigram lexical path needs no
  network) when the embed exceeds it. Recommend the budget+lexical-fallback first (simpler, bounded).

## Tests
- `runTurn`/system-block: a multi-iteration turn builds the system block **once** (spy the renderer);
  a mid-turn `remember` that changes core/L3 forces exactly one re-render; the block stays byte-stable
  otherwise (extends the existing byte-stable-prompt test).
- `trace/store.test.ts`: retention drops old rows past the window; `listTurns` is bounded; per-turn cap
  still holds.
- `recall.test.ts`: `listL2` is called with `limit:500` (spy); `content_hash` is read from the column,
  not recomputed (assert the hasher isn't called when the column is present); results identical to
  pre-change (golden).
- migration test: `content_hash` backfill is idempotent; schema version bumped.
- P1: with `LUNA_RECALL_ASYNC=1`, `open_stream` is not blocked on `retrieve` (order assertion); the
  lexical-fallback path returns within budget when embed is stubbed slow.
- `tsc` clean; `bun test` green.

## What this version explicitly does NOT include
No change to `history_json` persistence (A3/P2 → v0.16.2). No `vec0` decision (D1 → v0.16.2). No
text-mode removal (D2 → v0.16.2). No window-size change (→ Initiative 10) — A1/A2 just make the *current*
window cheap.

## Risk
**Low–Medium.** A1/A4 are pure recompute removal — the dirty-flag is the only correctness-sensitive bit
(get it wrong and a mid-turn memory edit isn't reflected; the test pins exactly that). A2 adds a
migration (must be versioned + idempotent backfill — the rewrite's anti-drift rule). P1 is the only one
that changes *when* recall runs — hence its own flag and the golden test that recall hits are unchanged;
the fallback must never silently drop recall quality without the budget actually being exceeded.

## Acceptance criteria
- [ ] System block built once/turn (re-rendered only on memory change); L1 contract built once; tested.
- [ ] `traces` retention window + bounded viewer query; tested.
- [ ] `recall` reads ≤500 L2 rows and reuses a persisted `content_hash` column; golden recall output
      unchanged; migration versioned + idempotent.
- [ ] `LUNA_RECALL_ASYNC` takes the query embed off the first-token path; flipped on after a latency
      measurement; recall hits unchanged.
- [ ] `tsc` clean, `bun test` green, DEVELOPMENT.md v0.16.1 entry.

## Notes for v0.16.2 (don't foreclose)
The `content_hash` column added here is also what an incremental `history_json` (A3) and a future
`vec0`-KNN wiring (D1) can lean on. Keep the recall candidate-assembly seam (`collectCandidates`) clean
— Initiative 10 will add **diaries** as a candidate source there.
