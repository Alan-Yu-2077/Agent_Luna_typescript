**作者：** Alan（项目所有者）

关于 L1 活跃窗口大小与日记注入的部分，以及 v0.4.1 计划里的窗口默认值。代码须向本文件对齐，而非相反。

## 结论

**当前的上下文太薄，由它构成的记忆系统记得太短——系统注入太少、活跃上下文窗口太小，完全不符合我本人的设计。**

我的设计是：

1. **L1 活跃上下文 = 100–150 轮的震荡压缩窗口**（不是十几轮，更不是按"消息条数"计）。
2. **把日记 / 周记的归纳注入上下文**（系统级），让她拥有跨天、跨周的长程记忆，而不是只记得眼前几轮。

现在两条都没做到。下面是核实过的代码事实。

---

## 代码现状 vs 我的设计

### 偏离一：活跃窗口缩了约 15–30 倍，单位还被悄悄换掉

- 代码现值：`KEEP_MSGS = 24`、`FOLD_MIN_BATCH_MSGS = 12`，折叠阈值 = 36
  ([`packages/server/src/memory/l1Window.ts:6-7,46`](packages/server/src/memory/l1Window.ts))。
- **单位是"消息条数"，不是"轮"。** 一个回合向 `history` 追加的消息见
  [`runTurn.ts:173,239,372`](packages/server/src/turn/runTurn.ts)：1 条用户输入 + 每轮工具迭代各 1 条
  assistant + 1 条 tool_result。所以一个普通 message-mode 回合 ≈ **4–6 条消息**。
- 换算：`KEEP_MSGS=24` ≈ **只有 4–9 轮逐字**。我的设计是 **100–150 轮** → **当前窗口小了约 15–30 倍**。
- 连 v0.4.1 的计划本身写的都是 `LUNA_L1_RECENT_TURNS` 默认"12 pairs"
  ([v0.4.1 计划](roadmap/memory-dream-substrate-2026-06/v0.4.1-l1-rolling-window.md))——也就是说**代码忠实实现了一个本就缩水的计划**，且实现时把变量从"轮"(`RECENT_TURNS`) 改成了"条"(`KEEP_MSGS`)。

> 我要的是 **100–150 轮震荡压缩**。现在是 ~一手之数的逐字 + 一段有损滚动摘要——记忆深度差了一个数量级。

### 偏离二：日记 / 周记从不注入，且被降级成了"只写不读"

- **零注入路径**（已全仓库核实）：系统块 [`renderCoreBlock`](packages/server/src/memory/renderCoreBlock.ts)
  不读日记；活跃窗口 [`buildActiveContext`](packages/server/src/memory/l1Window.ts) 不含日记；召回候选集
  [`collectCandidates`](packages/server/src/memory/recall/recall.ts)（`recall.ts:111`）只收 L2 + L3，**排除日记**；
  没有任何工具读日记。`diaries` 表的读者**只有做梦自身**（把天日记汇总成周日记）。
- **是被设计决策砍掉的，不是编码漏的**：LD #12 把日记重新定义为"住在 L2 的 dream 输出，不是一层"，并明确只把
  **core 散文**列为 always-injected——日记不在其列。
- **连归档都没当成**：`rag_refresh` 把日记 embed 进了缓存（[`cycle.ts:214`](packages/server/src/dream/cycle.ts)），
  但召回候选集不含日记 → **那些向量永远检索不到**，纯死功。
- **月度日记从不生成**：`diaryPrompt` 支持 `'month'`，但 [`cycle.ts`](packages/server/src/dream/cycle.ts) 只写
  `'day'` / `'week'`。

> 我要的是把**日记 / 周记的归纳注入上下文**。现在它们被写进 SQLite 后，对她在对话里"知道什么"的贡献是**零**。

### 结果：当前进入上下文的"记忆"实际只有这些

- ~4–9 轮逐字（L1 窗口）
- 一段有损的滚动摘要（`rolling_summary`，无结构、会随折叠增长）
- 召回命中（仅最近 500 轮 L2 + L3 事实，`RETRIEVAL_K=12`，每条截断 300 字）
- core 散文（self-state + relationship）

**缺失的，正是我设计里承载长程记忆的那一层——周期性的日记 / 周记归纳。** 所以她记得"刚才"，记得一些离散事实，
但没有"这几天、这几周发生了什么"的连续叙事记忆。这就是"记忆太短"的根因。

## 我的设计（更正后的目标，须对齐）

| 维度 | 当前 | 我的设计 |
|---|---|---|
| L1 活跃窗口 | 24 条消息（~4–9 轮） | **100–150 轮震荡压缩**，单位按"轮" |
| 日记 / 周记 | 写了但从不注入（死功） | **归纳后系统注入**，构成长程记忆 |
| 月度日记 | 从不生成 | 纳入（与日 / 周一致） |

更正方向（实现细节后续单独落地，本 PR 只锁定意图）：

1. **窗口**：把 L1 恢复到 **100–150 轮**的震荡压缩，单位改回"轮"（`LUNA_L1_RECENT_TURNS`），并据此重新权衡与"速度"目标的取舍——大窗口是我的设计要求，速度优化不得以牺牲记忆深度为前提默默实现。
2. **日记注入**：把日 / 周（及月）日记的归纳接回**注入路径**（进系统块或召回候选），让 `rag_refresh` 已经算好的嵌入真正被用上。
3. **决策更正**：相应修订 REWRITE_CONTEXT 的 **LD #12**——日记不再只是"L2 归档输出"，而是**需要注入上下文的长程记忆层**；L1 窗口大小写明为 100–150 轮。
